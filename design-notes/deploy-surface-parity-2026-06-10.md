## Question

What should the deploy surface on `pdpp.dev/reference` look like so that Railway, Fly.io, and Docker targets have comparable cognitive load for a new operator? Specifically:
1. Can Fly be a button comparable to the Railway button?
2. Can Docker be as easy and low-cognitive-load as Railway/Fly?

---

## Context

### Current state

| Target | Surface | Cognitive load |
|--------|---------|---------------|
| Railway | `[![Deploy on Railway](...)](https://railway.com/new/template/pdpp-core-template-source)` — true one-click, provisions Postgres, prompts for `PDPP_OWNER_PASSWORD` | Very low |
| Fly.io | A `fly launch` command block in `deploy/flyio/README.md` — operator must have `flyctl` installed, must have billing set up, runs a ~10-flag command | Medium–high |
| Docker | `docker compose --env-file .env.docker pull && docker compose --env-file .env.docker up -d` after filling `.env.docker` — requires cloning the repo, filling an env file, understanding compose | High |

### Repo layout facts relevant to each target

**Railway:**
Already live. Published template code `pdpp-core-template-source`. Button markup in `deploy/railway/template.md`. Image: `ghcr.io/vana-com/pdpp/railway-core:<tag>`. Single-container `platform-core`/`railway-core` Dockerfile stage runs console on `$PORT` + AS/RS on loopback. Postgres via Railway plugin. Template generates `PDPP_CREDENTIAL_ENCRYPTION_KEY` automatically; operator only sets `PDPP_OWNER_PASSWORD`.

**Fly.io:**
- `fly.toml` lives at `deploy/flyio/fly.toml` (not repo root).
- `fly launch --config <path>` supports this — no structural problem.
- The `platform-core` Dockerfile target is the same supervisor image used by Railway. It accepts `DATABASE_URL` (injected by `fly launch --db`) and falls back to SQLite when no database URL is present.
- There is no Fly.io "Launch on Fly.io" button product as of June 2026. The docs page for such a feature (fly.io/docs/launch/launch-button/) returns 404. No URL pattern exists to encode a one-click deploy.
- **Fly's equivalent of a button is a single `fly launch` command** (image-backed, ~10 flags). The repo already documents the honest fast path in `deploy/flyio/README.md`.

**Docker:**
- The current `docker-compose.yml` has `reference: depends_on: postgres: condition: service_healthy` — even though the runtime defaults to SQLite when no `DATABASE_URL` is set. The reference service starts Postgres unconditionally even when the operator wants SQLite.
- The GHCR image `ghcr.io/vana-com/pdpp/railway-core:main` can run standalone (no Postgres dependency) because `resolveStorageBackend` in `postgres-storage.js` falls back to `sqlite` when neither `PDPP_DATABASE_URL` nor `DATABASE_URL` is set. The `platform-core`/`railway-core` Dockerfile stage does not hardcode a database URL.
- The reference image does not generate or print an owner password on first boot. `PDPP_OWNER_PASSWORD` must be supplied by the operator. The server reads it from the env var; if absent, owner auth is disabled.
- `PDPP_REFERENCE_ORIGIN` defaults to `http://localhost:3002` in the compose file but is not defaulted in the standalone image. For a local single-container run it can be set to `http://localhost:3000`.
- The current `docker-compose.yml` does not have a minimal "SQLite-only, no Postgres" profile or override — the postgres service is always pulled in.

### Ideal Docker one-liner (proposed)

The target: one copy-paste command that requires no file download, no git clone, and no Docker Compose:

```sh
docker run -d \
  --name pdpp \
  -p 3000:3000 \
  -e PDPP_OWNER_PASSWORD="$(openssl rand -base64 24)" \
  -e PDPP_REFERENCE_ORIGIN="http://localhost:3000" \
  -v pdpp_data:/var/lib/pdpp \
  ghcr.io/vana-com/pdpp/railway-core:main
echo "Dashboard: http://localhost:3000/dashboard"
echo "Owner password: $PDPP_OWNER_PASSWORD"
```

This:
(a) pulls the public GHCR image `railway-core` (which already exists and is used by Railway/Fly)
(b) boots reference + console together via the existing `core-supervisor.mjs`
(c) uses SQLite (the runtime's default when no `DATABASE_URL` is supplied)
(d) persists data in a named volume `pdpp_data` at the image's default path `/var/lib/pdpp`
(e) prints dashboard URL + owner password from the shell (the password is generated by the shell command `$(openssl rand -base64 24)` and the operator sees it in the terminal)

### What exists vs. what is missing

| Requirement | Status | Gap |
|-------------|--------|-----|
| Public GHCR `railway-core` image | Exists (used by Railway template) | Confirm image is publicly pullable (anonymous) before presenting on docs page |
| SQLite fallback with no DATABASE_URL | Exists in `resolveStorageBackend` | No gap in code; just not exercised or documented for the standalone case |
| Console + reference in one container | Exists (`platform-core`/`railway-core` stage + `core-supervisor.mjs`) | No gap |
| Named volume persisting to `/var/lib/pdpp` | Exists — `PDPP_DB_PATH` defaults to `/var/lib/pdpp/pdpp.sqlite` | Confirm default path is within an easily mountable, non-ephemeral dir |
| First-boot generated + printed owner password | **Missing** | The image expects `PDPP_OWNER_PASSWORD` to be pre-supplied; it does not generate one. For the one-liner, the operator generates it locally with `$(openssl rand -base64 24)`. This is fine — the shell expansion handles it cleanly, but the operator must record the password. An alternative is an entrypoint that generates and prints one if the env var is absent. |
| `PDPP_REFERENCE_ORIGIN` default for localhost | Defaults to `http://localhost:3002` in compose, but not in the standalone image ENV | **Small gap:** add `ENV PDPP_REFERENCE_ORIGIN=http://localhost:3000` to the `railway-core`/`platform-core` Dockerfile stage so a bare `docker run -p 3000:3000` works without -e override. Or document the `-e` flag in the copy-paste command. |
| Compose "SQLite-only" minimal profile | Missing | Current `docker-compose.yml` always brings up Postgres. A `docker-compose.sqlite.yml` override or profile would let operators `docker compose --profile sqlite up` without Postgres. |
| Stable URL for a hosted compose file | Missing | If a curl-then-compose pattern is wanted (e.g., for multi-container with SQLite + optional Postgres), a stable `https://pdpp.dev/docker-compose.yml` or CDN URL would be needed. Not required for the `docker run` one-liner path. |

### Effort sizing

| Change | Size | Notes |
|--------|------|-------|
| Add `PDPP_REFERENCE_ORIGIN=http://localhost:3000` default to `railway-core` stage in Dockerfile | Small (1 line) | Makes bare `docker run -p 3000:3000 ... railway-core:main` work without `-e PDPP_REFERENCE_ORIGIN` |
| Entrypoint that generates + prints `PDPP_OWNER_PASSWORD` on first boot if absent | Small–Medium | Adds UX polish. Requires changing the CMD to a shell script that checks the env, generates if missing, stores in volume, prints to stdout. Touches `Dockerfile` + new `entrypoint.sh`. Security note: generated password must be persisted to the volume so restarts do not generate a new one; alternatively, print once and require operator to record it (like Umami's hardcoded default pattern). |
| Add a `docker-compose.sqlite.yml` minimal override | Small | No Postgres, SQLite volume only. Reduces compose cognitive load for operators who don't want Postgres. |
| Host a stable `docker-compose.yml` URL at a CDN / pdpp.dev path | Medium | Requires server-side routing or CDN config; allows `curl | compose up` pattern without git clone. |

## Current Leaning

**Fly:** Do not present a button. The honest surface is a `fly launch` command block with a one-line copy-paste. It is already documented in `deploy/flyio/README.md`. The reference page (`pdpp.dev/reference`) should show this as a "Deploy with Fly.io" code block — similar cognitive load to a button for developers who already have `flyctl`. Do not fabricate a button URL that doesn't exist.

**Docker:** The `docker run` one-liner (Small + Small–Medium effort) is the right target. The `railway-core` image already has everything needed. The main gaps are:
1. **(Small) Dockerfile default for `PDPP_REFERENCE_ORIGIN`** so the operator does not need to pass it.
2. **(Documentation only, zero code)** Presenting the `docker run` command prominently on the reference page with `$(openssl rand -base64 24)` inline password generation.
3. **(Optional, Small–Medium)** An entrypoint that auto-generates and prints the password if absent, for maximum UX parity with Railway's auto-generated `PDPP_CREDENTIAL_ENCRYPTION_KEY`.

The `docker-compose.sqlite.yml` override is a nice-to-have for operators who want to start simple and upgrade to Postgres later, but it is not required for the one-liner goal.
