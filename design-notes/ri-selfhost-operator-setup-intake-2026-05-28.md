## Question

What are the 2026 best-practice patterns for self-hosting one-command Docker apps—including RunPod templates, tunnel/public-origin setup, secret generation, and post-install admin onboarding—and what do those patterns reveal about the gaps between PDPP's current reference deployment story and what a friend or r/selfhosted reader expects?

This note is a prior-art synthesis and recommendation layer above two companion notes: the self-host SLVP decision note (`selfhost-runpod-onboarding-slvp-2026-05-27.md`) and the operator management capability map (`ri-operator-selfhost-management-map-2026-05-28.md`). It does not re-cover their ground; it adds what those notes lack: external prior art, a wizard-ownership framework, an essential/incidental classification of operator-management UI features, and a concrete recommendation for the smallest correct-by-construction next tranche.

## Context

The reference implementation ships Docker images and a Compose file. `add-selfhost-onboarding-slvp` is implemented: the quick-start doc (`docs/operator/selfhost-quickstart.md`) and the five-row deployment readiness panel are both done. The capability map identifies six lanes and recommends shipping five active changes in dependency order before designing more.

The gap this note addresses: neither companion note includes systematic prior art on what the r/selfhosted community expects in 2026, nor does either articulate a "wizard ownership" principle that answers the question "what is the minimum that must work before handing an instance to a friend, and who owns that surface?"

The primary audience for this intake is the reference implementation owner making triage decisions about what to implement next, not the r/selfhosted operator who will read `selfhost-quickstart.md`.

## Stakes

- A self-hosted deployment with invisible misconfiguration is a support call waiting to happen. Prior art shows the "missed required env var" class of failure is the most common first-boot failure mode across all self-hosted apps.
- Over-building the management UI (credential vaults, in-dashboard env edit) risks introducing durable contract changes under an "onboarding polish" label, contradicting the standing constraint from both companion notes.
- The secret-generation gap is the clearest unaddressed failure mode, and it is also low-cost to fix.
- Misclassifying the RunPod target (Hub = serverless, not persistent service) will waste a lane on the wrong artifact.

## Prior Art

### One-command Docker apps: what the ecosystem expects

The 2026 r/selfhosted ecosystem has converged on Docker Compose as the gold standard for multi-container applications. The target audience expects:

- A single `docker compose up -d` after populating a `.env` file (or a minimal copy of `.env.example`).
- A small number of required env vars (5–10 max) to bootstrap; source-specific credentials come later.
- Auto-renewing HTTPS via a reverse proxy (Caddy, Nginx Proxy Manager) or a tunnel (Cloudflare Tunnel).
- An in-browser experience immediately after first boot that either confirms the app is working or walks through remaining setup.

Sources: Immich, Jellyfin, Plausible Docker guides; Docker's own self-hosted app directory; Tipi one-click catalog.

PDPP's compose file ships with a ~290-line `.env.docker.example`. The r/selfhosted expectation is 5–10 required vars on first boot; the rest are optional.

### Secret generation

The Supabase Docker self-hosting pattern ships a `generate-secrets.sh` that auto-populates `SECRET_KEY_BASE`, `VAULT_ENC_KEY`, `PG_META_CRYPTO_KEY`, and `LOGFLARE_PUBLIC_ACCESS_TOKEN` into `.env` via `openssl rand`. The n8n self-hosting guide instructs users to run `openssl rand -hex 32` twice and paste into `.env`. Both explicitly warn: "never start with example or placeholder secrets."

PDPP requires at minimum:

- `PDPP_OWNER_PASSWORD` — the owner gate. Today: set manually or left blank (which the readiness panel surfaces as an error row).
- `VAPID_PUBLIC_KEY` + `VAPID_PRIVATE_KEY` for Web Push. Today: generated by `npx web-push generate-vapid-keys` and pasted manually.
- `SESSION_SECRET` for dashboard sessions. Today: set manually.

These three are the PDPP analog of Supabase's `generate-secrets.sh` scope. A one-command `scripts/generate-secrets.sh` that emits a `secrets.env` file (or writes directly to `.env.docker`) would close this gap without touching any protocol surface, control plane, or durable contract.

Sources: [Supabase self-hosting Docker](https://supabase.com/docs/guides/self-hosting/docker), [n8n self-host guide 2026](https://dev.to/jangwook_kim_e31e7291ad98/how-to-self-host-n8n-with-docker-ai-workflow-automation-guide-2026-3lec), [Docker Compose secrets](https://docs.docker.com/compose/how-tos/use-secrets/).

### Tunnel and public-origin setup

In 2026, Cloudflare Tunnel (`cloudflared`) is the dominant "get a public HTTPS URL without port forwarding" primitive. It:

- Runs as a Docker container added to the existing Compose stack (one extra service, one env var: the tunnel token from the Cloudflare dashboard).
- Provides a stable `*.trycloudflare.com` or custom-domain URL.
- Handles TLS termination at Cloudflare's edge.
- Is free and remotely managed (config lives in the Cloudflare dashboard, not a local `config.yml`).

For PDPP, Cloudflare Tunnel directly addresses the `PDPP_REFERENCE_ORIGIN` mismatch class of problem: the operator gets a stable URL before they set env vars, so origin and URL are consistent. Lane B (RunPod Pod) operators already get this from `*.proxy.runpod.net`. Lane A (Docker host) operators without a domain or open ports have no equivalent today—Cloudflare Tunnel is the canonical fill for that gap.

Trade-offs: Cloudflare terminates TLS and can inspect traffic; has a 100 MB upload limit on the free tier; TOS prohibits media serving via tunnel. None of these apply to the PDPP SLVP use case. Tailscale Funnel is the private-mesh alternative for operators sharing their instance with a small named set of users.

Sources: [Cloudflare Tunnel in 2026 (DEV)](https://dev.to/recca0120/cloudflare-tunnel-in-2026-expose-localhost-without-opening-ports-or-buying-an-ip-32l5), [selfhosting.sh Cloudflare Tunnel with Docker](https://selfhosting.sh/apps/cloudflare-tunnel/), [Cloudflare Tunnel vs ngrok vs Tailscale (DEV)](https://dev.to/mechcloud_academy/cloudflare-tunnel-vs-ngrok-vs-tailscale-choosing-the-right-secure-tunneling-solution-4inm).

The `selfhost-quickstart.md` mentions "CNAME-via-Cloudflare" as a vanity-domain escape hatch but does not explain Cloudflare Tunnel as a no-port-forwarding option. This is a doc gap, not an implementation gap.

### Post-install admin onboarding: Gitea and Coolify patterns

**Gitea:** First browser visit triggers a wizard that covers database type, network configuration (base URL, SSH port), and admin account creation in one page. The wizard exhausts required setup; optional settings (email server, storage drivers) go to admin panel post-install. After the wizard, the recommended next step is to set `GITEA__service__DISABLE_REGISTRATION=true` and redeploy—documented as a one-liner.

**Coolify:** Multi-step onboarding: create admin account → select server type (localhost) → SSH key verification → default project creation. Post-onboarding: set instance URL in Settings; disable public registration. Coolify also auto-provisions Let's Encrypt for deployed apps. Install is one curl-piped script that installs Docker, generates SSH keys, and starts Coolify containers.

Key pattern: both apps front-load "you must set this before use" into a browser wizard rather than relying on the operator to read docs before first boot. The wizard prevents access to the app shell until required setup is done.

PDPP's deployment readiness panel (`add-selfhost-onboarding-slvp`) is reactive (diagnoses a running instance) rather than proactive (prevents misconfigured access). The distinction matters for audience: Gitea and Coolify do not let you past the wizard without setting a password; PDPP lets you start without one and then flags it. This is a deliberate SLVP scope choice—"diagnostic surfacing, not a new control plane." The proactive gate (wizard that blocks `/dashboard` until password is set) is the correct next design but is out of scope for the current tranche.

Sources: [How to Self-Host Gitea (Localtonet)](https://localtonet.com/blog/how-to-self-host-gitea), [How to Install Coolify on VPS (MassiveGRID)](https://massivegrid.com/blog/how-to-install-coolify-on-vps/), [I Compared 6 Platforms for Deploying Self-Hosted Apps in 2026 (DEV)](https://dev.to/vikasprogrammer/i-compared-6-platforms-for-deploying-self-hosted-apps-in-2026-3j8).

### RunPod Hub: serverless vs persistent service — critical distinction

**The RunPod Hub is designed for serverless AI workers, not persistent services.** Hub deployment requires:

- `handler.py` implementing the serverless endpoint interface.
- `.runpod/hub.json` + `.runpod/tests.json` defining metadata and test cases.
- A GitHub Release to trigger the automated build pipeline.
- One container per Hub listing.

PDPP's reference implementation is a persistent service (AS/RS + dashboard), not a stateless serverless function. The Hub model is not the right target. The correct RunPod surface for PDPP is a **Pod template** (persistent Pod, not serverless)—a saved Docker image + startup command in the RunPod Pod creation UI. Pod templates can be community-shared independently of the Hub indexing pipeline.

The "deferred RunPod Hub template" named in both companion notes should be reframed in the decision log: it is a **RunPod Pod template with a single-container `pdpp-all-in-one` image**, not a Hub serverless worker listing. Building this requires a new Docker image shape (process supervision, single startup script that auto-generates missing secrets, SQLite-only Postgres-optional default) plus a release cadence. Effort is L; correctly classified as deferred.

Sources: [RunPod Hub deep dive](https://www.runpod.io/blog/deep-dive-runpod-hub), [Manage Pod templates](https://docs.runpod.io/pods/templates/manage-templates), [RunPod containers GitHub](https://github.com/runpod/containers).

### Backup and export

The 2026 self-host backup pattern for Postgres + Docker volumes:

```sh
# Database
docker exec pdpp-postgres pg_dump -U pdpp pdpp | gzip > pdpp-db-$(date +%Y%m%d).sql.gz
# Runtime state volume
docker run --rm --volumes-from pdpp-reference \
  alpine tar czf - /app/home > pdpp-home-$(date +%Y%m%d).tar.gz
```

Optional: Restic sidecar container for automated versioned offsite backup. PDPP's quick-start covers this pattern. The management map explicitly defers backup/export UI—correct for the r/selfhosted audience, who are comfortable with CLI-based backup. A dashboard UI that wraps `pg_dump` is incidental, not essential.

Sources: [Restic Docker Volume Backup: Compose Pattern (servercrate)](https://servercrate.net/restic-docker-backup/), [How to Back Up and Restore Docker Volumes (oneuptime)](https://oneuptime.com/blog/post/2026-01-06-docker-volume-backup-restore/view), [PostgreSQL Docker backup strategies (DEV)](https://dev.to/piteradyson/postgresql-docker-backup-strategies-how-to-backup-postgresql-running-in-docker-containers-1bla).

## Wizard Ownership Framework

Based on the prior-art survey, the 2026 self-host ecosystem has converged on this ownership model:

| Layer | Owns | 2026 examples |
|---|---|---|
| Shell setup script | Auto-generate substrate secrets (password, VAPID keys, session secret) before first boot | Supabase `generate-secrets.sh`, n8n `openssl rand` instructions |
| First-boot browser wizard | Required configuration that cannot be safely deferred (admin account, database selection, base URL) | Gitea install wizard, Coolify onboarding flow |
| Dashboard readiness panel | Ongoing verification; surfaces misconfiguration in a running instance | PDPP `DeploymentReadinessPanel` (5 rows), Gatus, Kener |
| Dashboard settings / secondary config | Optional capabilities (email, HTTPS cert, SSH key verification) | Coolify Settings, Gitea admin panel |
| CLI | Power-user diagnostics, enrollment, local collector operations | PDPP `pdpp ref`, `pdpp collector` |
| Docs | Substrate-specific guidance, connector credentials, advanced topology | PDPP `selfhost-quickstart.md`, `hosted-mcp-setup.md` |

### PDPP's current posture vs. the framework

| Layer | Current state | Gap |
|---|---|---|
| Shell setup script | Not implemented | `generate-secrets.sh` missing; operators hand-generate `PDPP_OWNER_PASSWORD`, VAPID keys, `SESSION_SECRET`. The most consistent gap between PDPP and ecosystem expectations. |
| First-boot browser wizard | Not implemented | Readiness panel is the current substitute (reactive, diagnostic). A true first-boot wizard (proactive, blocks `/dashboard` until setup is complete) is deferred. |
| Dashboard readiness panel | Implemented (5 rows, `add-selfhost-onboarding-slvp`) | A Tier-2 row set (Web Push VAPID, n.eko, Postgres, host browser bridge, `PDPP_TRUSTED_HOSTS`) would extend coverage. Management map recommendation 4—optional. |
| Dashboard settings / config | Partial | Per-connector configuration-readiness rollup (env + manifest + coverage synthesis) is the named next slice per management map Lane 3. Not yet implemented. Needs design note → OpenSpec. |
| CLI | `pdpp ref` + `pdpp collector` implemented | Reasonable for SLVP. Mutation coverage (schedule pause/resume, connection display-name patch) is incomplete but not blocking the spin-up-to-grant journey. |
| Docs | `selfhost-quickstart.md` and `hosted-mcp-setup.md` implemented | Cloudflare Tunnel option added to Lane A. RunPod Pod template framing corrected in quickstart and companion design notes (2026-05-29, `ri-selfhost-phase15-secrets-docs`). |

### Notes on specific features

**Connection management.** Today connections are env-var-configured. The dashboard does not synthesize "Gmail wants `GMAIL_APP_PASSWORD`; you have set it; the connector is `proven_working`" across env + manifest + coverage. This is the largest mid-term gap in the self-host experience. It requires the per-connector config-readiness rollup (management map Lane 3), which needs its own design note before implementation.

**n.eko / browser surfaces.** For the r/selfhosted audience, n.eko is a connector-specific advanced feature, not part of the "spin up and permission Claude" baseline journey. The management map correctly defers it. Surface lease management (`add-neko-browser-surface-leases`) is active and tracks this capability.

**Schedules needing attention.** The "something needs your attention" surface is the most common reason a self-hosted user returns to their instance's dashboard—a pattern seen in Miniflux, n8n, and similar self-hosted workflow tools. If the scheduler runs but there is no attention surface, the operator will not know to return. The `define-schedule-manual-attention-policy` + `wire-reference-scheduler-loop` tranche is therefore essential for an ongoing-operations experience, even if it is not blocking the first connection and first grant.

**Backup and export UI.** The r/selfhosted audience tolerates CLI-based backup (`pg_dump` + volume tarball) when it is documented clearly. A dashboard UI that wraps these commands is incidental for this audience. Correctly deferred.

**Semantic index status.** Already surfaced in the existing `DeploymentDiagnosticsView` via `/_ref/deployment`. Not a gap; no action needed.

## Public Site vs Self-Hosted Operator Console

`public-site-vs-reference-server-split-2026-05-21.md` covers this exhaustively. For self-host framing, the key operator-facing consequence is:

- When running Docker Compose locally, the browser should reach the **operator console** (`apps/console`), not the public marketing site (`apps/site`). Today both are served from `apps/web`.
- The reference server's `/` discovery JSON is correct AS/RS behavior and should not be confused with "the PDPP home page."
- `split-public-site-and-operator-console` (active) resolves this. Until it lands, the quick-start should link operators directly to `/dashboard`, not `/`, which is the current behavior.
- For an operator sharing their instance with a friend, the `/dashboard` URL is the entry point; `/docs` and `/reference` are informational and irrelevant to first-boot setup.

The site/console split does not block the r/selfhosted SLVP journey, but it resolves the "I opened localhost:3002 and got marketing copy" confusion class. Treat it as a parallel lane, not a prerequisite for the next tranche.

## Current Leaning

### Smallest correct-by-construction tranche

The management map recommends shipping five active changes in dependency order. This note adds two items: one pre-flight (new) and one doc gap (in existing doc):

**Item 0 — Secret generation script (new, not in any active OpenSpec change):**

Add `scripts/generate-secrets.sh` that auto-generates `PDPP_OWNER_PASSWORD` (if absent), `SESSION_SECRET`, and VAPID key pair via `openssl rand` and `npx web-push generate-vapid-keys`, then writes them to `.env.docker` or a `secrets.env` include file. The quick-start's step 2 becomes "run `./scripts/generate-secrets.sh`" instead of "edit the file and set each variable manually."

This is one day of work. It does not touch any protocol surface, OpenSpec requirement, or durable contract. It can ship as a follow-up commit under `add-selfhost-onboarding-slvp` or as a standalone fix commit.

**Item 0b — Cloudflare Tunnel in quickstart (doc-only, no new OpenSpec):**

Add a "no domain, no open ports" sub-section to Lane A of `selfhost-quickstart.md` explaining that adding `cloudflared` as a Compose service with a tunnel token gives a stable public HTTPS URL and ensures `PDPP_REFERENCE_ORIGIN` is set correctly before `docker compose up`. This is a paragraph and a Compose snippet; no implementation change required.

**Items 1–5 — From the management map (ship in dependency order):**

1. Security/posture queue: `harden-reference-auth-surfaces` → `gate-ref-reads-when-owner-auth-enabled` → `honor-csrf-exemption-for-bff-device-flow` → `polish-dashboard-auth-pwa-ux` → `dcr-per-owner-token-with-revoke`.
2. `expose-connection-identity-on-public-read` — unblocks per-connection UI in Lane 3.
3. `wire-reference-scheduler-loop` — loop wiring, unblocks attention surface.
4. `define-schedule-manual-attention-policy` — attention contract, unblocks attention triage UI.

**What this tranche explicitly does not include:**

- First-boot browser wizard (proactive gate on `/dashboard`): deferred. Worth planning when the target audience expands beyond r/selfhosted.
- Per-connector config-readiness rollup: deferred; needs its own design note → OpenSpec first.
- In-dashboard credential management UI: deferred; separate OpenSpec change.
- RunPod Pod template / single-container image: deferred; separate design note + OpenSpec + new image shape.
- Tier-2 readiness rows: optional; not required for the coherent-deployment journey.

### Is a first-boot browser wizard worth building before the tranche above?

No. Prior art shows the wizard is most valuable for a non-technical audience that expects hand-holding. The r/selfhosted target audience is comfortable with a well-written quick-start and env-var setup, especially when secrets are auto-generated and a readiness panel confirms them. The generate-secrets.sh script + the existing readiness panel + the clear quick-start together constitute the PDPP equivalent of Gitea's install wizard for this audience.

A proactive wizard (blocking `/dashboard` access until password is set) is the correct design for a wider, less technical operator audience. It should be planned when that audience is explicitly in scope, but is not part of this tranche.

## Options

Three options for the next implementation focus, given the above:

**Option A — Security queue only (S, ~3–5 days).** Ship the five active security/posture changes in dependency order. Safe and contained. Leaves the generate-secrets.sh and Cloudflare Tunnel doc gaps open.

**Option B — Security queue + quick-start improvements (S–M, ~5–7 days).** Ship option A plus `generate-secrets.sh` and the Cloudflare Tunnel doc addition. This is the recommended tranche—it closes the "friend spins this up and fails on the first required env var" failure mode while staying in the existing operational footprint.

**Option C — Security queue + connection identity + scheduler loop (M, ~10–14 days).** Ship option B plus `expose-connection-identity-on-public-read` and `wire-reference-scheduler-loop`. This is the tranche that makes the ongoing-operations experience coherent: the operator can see connections and schedules working after first setup. The right next tranche after option B.

**Recommendation: option B first, then option C.**

## Non-Goals

- Multi-tenant PDPP hosting. There is no PDPP-the-company hosted service.
- Auto-update / Watchtower. Operator-driven `docker compose pull && docker compose up -d` only.
- TLS at the PDPP layer. Cloudflare Tunnel / reverse proxy / `*.proxy.runpod.net` are the documented paths.
- Multi-operator RBAC. Single owner password is the correct-by-construction SLVP model.
- Kubernetes deployment. Out of scope for the r/selfhosted audience in this tranche.
- SaaS onboarding semantics ("sign up for PDPP"). Voice/framing guide prohibits this.
- RunPod Hub serverless listing. PDPP is a persistent service; Hub is for stateless AI workers.
