## 1. State of the UI in 10 sentences

1. PDPP already has a real design-system *foundation* that beats most early-stage products — a `pdpp-brand` package with oklch semantic tokens, a 4px spacing scale, a radius scale, a motion scale, a named 9-step type scale, and full dual-theme dark-mode coverage — so at first glance the product reads as coherent (`component-inventory.md` §1; `critique-system-consistency.md` Preamble).
2. The shared layer is thinner than it looks: only `operator-ui` is genuinely shared, while 20 files (12 `ui/` primitives, 5 `pdpp/` components, 3 theme files) are copied byte-for-byte between `apps/console` and `apps/site`, and `consent-card.tsx` has already silently diverged 76 lines (site has multi-connection support, console does not) — proving the duplication will keep drifting (`component-inventory.md` §3; `critique-system-consistency.md` cross-cutting).
3. The single most damaging systemic gap is that **connection health — the product's primary diagnostic concept — renders three different ways across the three surfaces that show it**: a `StatusBadge` chip on the Sources list, plain header text on the connection detail page, and nothing but an inferred failure-count callout on the sandbox overview (`critique-system-consistency.md` V2/V3 + cross-cutting P0).
4. There is **no canonical `--status-*` surface-token tier** in the brand package, so every health surface (badges, dots, failure chips, callout borders) is assembled ad-hoc from whatever nearby token the author reached for (`--success-wash`, `--human-wash`, raw oklch), which is the root cause of the visible status fragmentation (`ux-debt.md` #15; `critique-system-consistency.md` cross-cutting P0).
5. The Sources list — the operator's main triage surface — fails the 30-second-orientation test: 19 connectors render as a wall of equal-weight rows with stacked diagnostic badges, no left-anchored status column, and "Track only"/"Manual setup" disposition labels in warning-amber that make working connectors *look broken* (`critique-slvp-comparison.md` View 02 = 3/10; `critique-hierarchy-typography.md` View 2; verified in `02-sources-list-desktop.png`, a dense field of pink rows).
6. The Deployment Settings page is a genuine credibility risk — a ~6,400px single-column scroll with no section navigation that mixes critical settings (owner password, origin) with operational data dumps (a 71-row participation table, machines, raw env vars including filesystem paths) and an unguarded destructive "RESET" at the same visual weight as informational rows (`critique-slvp-comparison.md` View 08 = 2/10; verified in `08-deployment-settings-desktop.jpeg`).
7. The Grants page commits a semantic error visible to any standards reviewer: it borrows the run-lifecycle vocabulary (`SUCCEEDED`/`FAILED`) for grant *lifecycle* states (which should read issued/active/revoked/denied), and pairs it with opaque scope identifiers (`package-gqkg_…`) and full inline `chatgpt.com` redirect URIs that overwhelm every row (`critique-system-consistency.md` V7 + cross-cutting; `critique-slvp-comparison.md` View 07 = 3/10; verified in `07-grants-desktop.jpeg`).
8. Mobile is the sharpest credibility cliff: at 390px the Sources list collapses to an ~9,000px-tall wall of 200–350px stacked cards where desktop badge clusters wrap line-by-line and health badges/CTAs are clipped or absent — there is no mobile-specific card layout anywhere, just a shrunk desktop table (`critique-system-consistency.md` V12 = 2/10; `critique-slvp-comparison.md` View 12 = 3/10; verified in `12-sources-list-mobile-390.jpeg`).
9. The public site's landing hero is by contrast the product's *strongest* asset and the nearest thing to the SLVP bar — a clean two-column "consent surface vs. grant+response" proof panel with an interactive consent card and a `continuous/single_use` toggle — but it is undercut by inline-style token leakage in `reference-app.tsx` (156 inline styles drive the field chips, temperature colors, and card accents outside the token system), an ambiguous floating "Grant: idle" pill that collides with the "Allow access" button, a still-static field-projection hero, and a `localhost:3002` URL leaking into production on the Reference page (`critique-slvp-comparison.md` Views A/B/C/D; `ux-debt.md` #6; `site/index.md`; verified in `landing-desktop-hero-1440.png`).
10. The defined 9-step type scale exists but is applied inconsistently — section headers on the Dashboard overview blend into body text while the connection-detail page steps them up correctly — so the single highest-ROI move across the whole console is enforcing the existing scale and shipping the missing `--status-*` tokens, *before* any structural redesign (`critique-hierarchy-typography.md` CX.1 + 6-line summary; `critique-system-consistency.md` 6-line summary).

## 2. The design-system verdict

**What exists (real, keep):** A legitimate token foundation in `pdpp-brand/base.css` — oklch semantic colors, extended semantic washes, a radius scale, a 4px spacing scale, a motion/easing scale, a layout-dimension scale, a `--numeric` (tabular-nums) feature token, a named 9-step type scale (`pdpp-display-lg` … `pdpp-eyebrow`), and complete dark-theme re-declarations. `operator-ui` exports a genuinely-used shared primitive set: layout/composition primitives (`PageHeader`, `Section`, `Toolbar`, `SplitLayout`, `DataList`, `Pager`, `MetaPill`, `StatusBadge`, `Callout`, `FilterSummary`), Base-UI-backed `ui/` wrappers (`button`, `dialog`, `input`, `select`), and shared components/views (`CommandPalette`, `EmptyState`, `RunRow`, `TimelineView`, `OverviewView`, `RecordsExplorerView`, status vocabularies). The status-badge color system (green/red/amber/grey) is the most successful single design element in the product (`critique-hierarchy-typography.md` CX.3).

**What leaks (fix at the source):**
- **The 20-file duplication layer** (`ui/` + `pdpp/` + `theme/` copied into both apps) — structural divergence already realized in `consent-card.tsx` (`component-inventory.md` §3).
- **Missing `--status-*` surface tier** — no `--status-{success,warning,danger,neutral}-{bg,fg}` pairs, so status surfaces are inconsistent everywhere (`ux-debt.md` #15).
- **Tokens not mapped into Tailwind theme** — the `tailwind.config` in each app does not extend `theme.colors/borderRadius/spacing` with the CSS-variable values, forcing pervasive `text-[color:var(--x)]` bracket syntax (209+ dimensional bracket values) and pushing authors toward inline styles; the Tailwind content scan also excludes `operator-ui`, so `dark:` utilities can't generate there and `base.css` carries CSS workarounds (`component-inventory.md` §1/§4b; `critique-system-consistency.md` cross-cutting P2).
- **Inline-style cluster** — ~220 production inline `style={}` props (excluding doc pages), concentrated in `pdpp/` (consent-card, grant-inspector, connector-card, stream-inventory) and `reference-app.tsx`; these components predate the utility layer and put dark-mode/responsive/hover styling out of reach of CSS (`component-inventory.md` §4c).
- **Vocabulary leakage** — run lifecycle (`SUCCEEDED`/`FAILED`) reused for grant lifecycle; coverage/freshness pills use a second badge family alongside `StatusBadge`; monospace applied inconsistently to technical identifiers (`critique-system-consistency.md` V7 + cross-cutting P1/P3).

**Keep / rebuild call per layer:**

| Layer | Verdict | Rationale |
|---|---|---|
| **Tokens** (`pdpp-brand/base.css`) | **KEEP + extend** | Foundation is strong; the only deltas are additive: ship the `--status-*` surface tier, add a `--z-overlay`/pill layer token, and map all tokens into both apps' Tailwind `theme.extend`. Do not rebuild — extend. |
| **Primitives** (`operator-ui/ui/*`, Base-UI wrappers) | **KEEP, consolidate, fill 2 gaps** | Base-UI coverage is good. Make `operator-ui` the *single* source and delete the 20 app-local copies; back the hand-rolled `CommandPalette` and `MobileDrawer` with headless primitives; add `Tabs`, `Toast`/`Notification`, and `StatusBadge` variants driven by the new status tokens (`component-inventory.md` §5). |
| **Components** (`operator-ui/components/*`, `pdpp/*`) | **REBUILD the `pdpp/` cluster; consolidate the rest** | The 4 `pdpp/` components are the worst hygiene offenders (inline-style heavy, duplicated, diverged) — rebuild them token-clean inside `operator-ui` and de-duplicate. `consent-card` is the flagship rebuild (see §4). Other shared components stay; promote app-local survivors (`connector-card`, `grant-inspector`, `spec-citation`, `stream-inventory`, `pdpp-logo`, theme files) into `operator-ui`. |
| **Views** (`operator-ui/views/*` + console route pages) | **RE-COMPOSE, not rebuild** | The view *content* is thorough; the failures are hierarchy/IA, not missing data. Sources list, Grants, Deployment, Runs list, and all mobile breakpoints need re-composition against a one-status-per-entity, left-anchored, scannable pattern with mobile-specific card layouts. The data layer (`lib/`, `ref-client`, `rs-client`) is untouched. |

**One-line system verdict:** the foundation is keep-and-extend, the duplication layer and `pdpp/` component cluster are rebuild, and every view is a re-composition against scannability — there is no need to throw away the token package or the primitive library.

## 4. Overhaul strategy recommendation

**Guiding principle:** build *outward from the token core*, landing each layer where it can't collide with the active `workstream/ri-*` console branches. The collision zone is concrete: live workstreams are editing Sources IA, static-secret lifecycle, browser-dashboard setup, deployment headroom, and the owner-journey harness — i.e. exactly the console *route pages*. So console *views* are sequenced **last**, and only after the shared layer they depend on is stable.

**Sequencing (3 phases of build, gated by a design-direction decision):**

- **Phase 1 — Foundation, in `pdpp-brand` + `operator-ui` first (zero console-route collision).** Ship changes #1–#5 and #12 from the ranking: the `--status-*` token tier, the Tailwind-theme mapping + content-scan fix, the 20-file de-duplication, the `GRANT_LIFECYCLE_VOCABULARY`, the type-scale enforcement helper, and the `Callout` `info`/`warning` tones. These all live in the brand package and `operator-ui`, which the `ri-*` branches consume but rarely edit — so this work merges cleanly underneath them and *improves* their surfaces for free. Rebuild the `pdpp/` component cluster (consent-card, grant-inspector, connector-card, stream-inventory) token-clean here. This phase is the load-bearing one: every later view fix becomes a re-composition rather than a redesign.

- **Phase 2 — Site / landing second (separate app, separate audience, low collision).** `apps/site` is edited far less by the active console workstreams, and it carries the highest external-credibility leverage (it is what the CEO shows investors and what engineers hit first). Ship #10 (production URL fix — do immediately, it's trivial and embarrassing), then #13 (rebuild `reference-app.tsx` on tokens, anchor the pill, animate the field-projection hero), plus the landing hero polish (subline weight, active-nav-pill color, CONSENT/GRANT eyebrow legibility, og:image). The flagship landing work depends on Phase-1 tokens but not on console views.

- **Phase 3 — Console views last (highest collision; sequence per-view as `ri-*` branches land).** Re-compose Sources (#6), connection detail health + diagnostics (#7, #14), Deployment section-nav (#8), Runs list (#11), mobile layouts (#9), and catalog/sandbox disambiguation (#15). **Coordinate per-view:** do not start a view's re-composition until its owning `workstream/ri-*` branch (e.g. `ri-sources-ia-add-account-v1` for Sources, `ri-deployment-headroom-v1` for Deployment, `ri-static-secret-visible-lifecycle-v1` for the connect flow) has merged or been explicitly parked — otherwise the overhaul rebases against a moving target. Because Phase 1 already shipped the tokens/vocabulary/primitives these views need, Phase 3 is composition work, which is faster and less merge-prone than it looks.

**What Phase 2 (design direction) must decide** — these are choices the synthesis cannot make for the owner; they gate the visual rebuild:
1. **Brand temperature & accent system** — formalize the copper-human / blue-protocol temperature coding (today it's a strong idea applied inline and incompletely on the landing nav) into named tokens, or retire it. This decision drives the consent-card, grant-card, and protocol-nav rebuilds.
2. **The single canonical health-status visual** — pick the one badge/dot/row-tint treatment that will represent connection health on every surface (the new `--status-*` tokens encode it, but the *shape* — pill vs. dot+label vs. left-border — is a design call).
3. **Density posture** — Stripe-dense data tables vs. Linear-airy cards. The console currently mixes both; the owner must pick a default so Sources/Runs/Grants/Deployment converge instead of each re-deciding.
4. **Mobile model** — bottom-tab bar (Linear) vs. hamburger-drawer (current), and the mobile card anatomy (one-status-one-CTA). This gates all of #9.
5. **Type & motion identity** — confirm Geist Variable / JetBrains Mono as the locked pairing and define the motion vocabulary for the animated landing (the tokens exist; the *direction* — restrained Vercel vs. expressive Stripe — is undecided).

**Flagship view candidates** (the views that should be designed first and to the highest finish, because they carry the most credibility for each audience):
- **Consent card** (landing + console) — already the best component in the product and the clearest articulation of the entire value prop; the rebuild target that investors and standards reviewers will judge the protocol by. *(Best for: standards reviewers, CEO/investors.)*
- **Connections (Sources) list** — the operator's home base and currently the worst-scoring core console surface; the before/after that engineers evaluating adoption will feel most. *(Best for: engineers evaluating adoption.)*
- **Run timeline** — the proof that the system is honest about what it did and when; the surface where "the system noticed and paused" reassurance lives. *(Best for: engineers, standards reviewers.)*
- **Landing hero** — the literal first impression for every audience and the nearest-to-bar asset; small investment, outsized return. *(Best for: CEO/investors first, everyone second.)*

## 5. Risks

- **Workstream collision (highest).** Console route pages are under active `ri-*` development; starting Phase-3 view re-compositions before those branches land guarantees rebases and possible regressions to in-flight onboarding/lifecycle work. *Mitigation:* the phasing above + per-view gating on the owning branch.
- **Tailwind-theme mapping is a quiet breaking change.** Mapping CSS vars into `theme.extend` changes how existing `text-[color:var(--x)]` and arbitrary utilities resolve; a careless migration can shift colors/spacing app-wide. *Mitigation:* introduce mapped utilities additively, migrate bracket usages in a separate pass, keep the brand package the single source of truth.
- **De-duplication can regress the app that was *ahead*.** `consent-card` diverged with the *site* copy ahead (multi-connection support); a naive "promote console's copy" loses that. *Mitigation:* reconcile to the superset before deleting either copy; add a test pinning multi-connection behavior.
- **Re-composition that loses data fidelity.** The console's honesty (unknown ≠ zero, explicit failure reasons, real run IDs) is a protocol value, not clutter; aggressive "simplify for scannability" passes risk hiding load-bearing facts. *Mitigation:* move detail into expansions/peeks, never delete it; keep the existing `lib/` data layer intact.
- **Design-direction churn stalls the build.** Phase-1 foundation can ship before the Phase-2 direction decisions, but the *visual* rebuild (consent card, landing, status-shape) cannot start until decisions 1–5 are made; leaving them open mid-build causes rework. *Mitigation:* gate the visual rebuild on a written design direction; let token/vocabulary/de-dup work proceed in parallel.
- **Mobile is a redesign, not a reflow.** The mobile failures (Sources, Dashboard) are structural; treating them as CSS breakpoint tweaks will reproduce the current ~9,000px stacked-card wall. *Mitigation:* design mobile card anatomy explicitly in Phase 2 (decision 4), build in Phase 3.
- **Screenshot blind spots.** Three console captures (run-timeline, explorer, and one connection-detail attempt) rendered the landing page due to a Playwright profile redirect, so timeline and explorer findings are inferred from code, not pixels (`console/index.md`; both critique files flag the gap). *Mitigation:* re-capture those views before finalizing the run-timeline and explorer flagship work.
- **Site captures used font fallback + accidental 390px frames.** Several "desktop" landing captures were actually 390px and most used `sans-serif` fallback, so type-size and nav-collapse judgments on the site carry a calibration caveat (`site/index.md`). *Mitigation:* re-verify landing typography and the mobile nav-collapse breakpoint against the real custom font before Phase-2 sign-off.
