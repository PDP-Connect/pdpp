**MEMORANDUM**
**TO:** The PDPP Authors / Core Contributors
**FROM:** ChatGPT
**DATE:** April 9, 2026
**SUBJECT:** Clarifying Enforcement Boundaries, Trust Assumptions, and Promise Surface in PDPP v0.1

### Executive Summary

This memo is intended as a complement to the existing architectural memo, not a rebuttal to it. The first memo is right to focus on projection-safe deltas, freshness, joint-data handling, scraping as polyfill, and day-two operational realities. The additional concern raised here is different: PDPP’s largest near-term landing risk is not that its architecture is incoherent, but that its strongest user-facing surface can imply broader enforceability than the protocol actually provides. PDPP’s core is clean: grants parameterize consent, the resource server enforces disclosure, and collection is a separate companion concern. But once the consent surface includes purpose, retention, and client-authored commitments alongside field/stream selection, readers can easily blur together “hard protocol constraints” and “soft or external commitments.” That ambiguity is the main thing this memo tries to close.

My conclusion is that PDPP should not remove purpose, retention, or client commitments. It should instead type them more explicitly. Today the spec already says that purpose-code consent properties are advisory except for `ai_training`, that `client_claims` are non-enforceable, that retention is not technically enforced, and that revocation stops future access only. Those are good and honest design choices. The problem is that they are not yet elevated enough in the protocol’s presentation, conformance model, and trust narrative.

### 1. The Core Protocol Is Coherent, but the Promise Surface Is Wider Than the Enforcement Surface

PDPP is strongest where it is narrowest. The normative core is the grant and the query API; collection is explicitly a companion mechanism. The record model is flat relational streams, and the resource server is responsible for enforcing stream membership, field projection, time range, and resource filters on every request. That part hangs together.

The problem starts when readers naturally generalize from that strong disclosure machinery to broader trust claims. In the current spec, “you can only get these fields from these streams” is hard protocol behavior. By contrast, “you will only use the data for this purpose,” “you will delete it after this duration,” and “these are my commitments” sit in a different category. The spec already admits this: retention is a policy commitment, `client_claims` are non-enforceable, and purpose-code consent properties are advisory except for `ai_training`. But the current structure still invites people to read these as one coherent enforcement layer when they are not.

This is not just a wording issue. It is an architectural one. PDPP enforces disclosure minimization very well. It does not, by itself, fully enforce downstream behavior. That distinction needs to become one of the spec’s main organizing principles.

### 2. PDPP Should Explicitly Divide Request/Grant Semantics into Three Classes

The cleanest fix is to add an explicit semantic split across request and grant fields.

First, there are **hard protocol constraints**: streams, fields, views, resources, time ranges, access mode, and revocation of future access. These are actually enforced by the authorization server and resource server.

Second, there are **structured policy declarations**: purpose code, retention, and similar machine-readable statements that matter for consent, audit, client policy, and trust registries, but are not generally self-enforcing at the protocol layer. The spec already effectively treats retention this way, and treats most purpose semantics this way too.

Third, there are **attributed soft commitments**: client-authored claims that may be important to the user but are not protocol facts. The spec already classifies `client_claims` this way.

I recommend making this tripartite distinction explicit in the field tables and prose. Add an “enforcement status” column. Do not force readers to infer it from scattered notes. If PDPP adopts more GDPP-style metadata, this becomes even more important, not less. The first memo is right that freshness and data-class metadata would improve regulator-readiness. But every additional policy field increases the risk that users will over-read the protocol’s guarantees unless enforcement status is made impossible to miss.

### 3. Consent UI Separation Should Be Normative, Not Merely a Good Product Instinct

The spec currently assigns the authorization server responsibility for consent UI, but “consent screen UX” is also listed as surface-specific and out of scope. That is too weak for one specific issue: the distinction between connector/manifest facts and client-authored claims. If PDPP wants to carry non-enforceable claims in a consent request, the authorization server should be normatively required to render them in a visibly different way from actual grant constraints and manifest-derived data descriptions.

In other words, the protocol should not only say that `client_claims` are non-enforceable. It should require that this be legible to the user. A conformant authorization server should not be allowed to flatten “requested fields” and “requester promises” into a single undifferentiated consent card. This is the clearest place where a modest amount of UI normativity would protect the integrity of the protocol’s semantics.

### 4. Revocation and Deletion Must Be Explicitly Separated

The current spec is honest here: revocation propagates through introspection, stops future access, and data already delivered to the client is governed by retention and legal obligations. The first memo is right that this will not match ordinary user expectations. Users will often experience “revoke” as meaning “make it go away.” The current protocol model does not provide that.

I therefore recommend that PDPP add a separate erasure or deletion signal rather than trying to overload revocation. This could remain a deferred extension, but the distinction should be stated prominently now: revocation is not deletion. Otherwise one of the most important consent-management verbs in the protocol will remain semantically slippery in practice.

### 5. Connector Trust Is Not an Edge Concern; It Is Part of PDPP’s Credibility Surface

The current security text is candid that malicious connectors can exfiltrate credentials and that formal connector trust is deferred. That candor is good, but the topic is still treated as more deferrable than it really is. In any Collection Profile deployment, connector provenance, isolation, and update policy are first-order security assumptions, not backend implementation details.

This is where the first memo’s OTA recommendation is directionally right but incomplete. The issue is not just maintenance rot. It is also whether the ecosystem can say, with a straight face, what it means to trust a connector at all. If PDPP intends third-party or community connectors to be part of the story, then connector signing, registry trust levels, sandbox expectations, and update provenance should become more central in the roadmap and threat model.

### 6. Some of the Protocol’s Cleanest Ideas Depend on Stronger Conformance Language

The spec makes a strong and valuable privacy claim for `changes_since`: the resource server should return only records whose grant-authorized projection changed, and hidden-field changes should not be inferable. That is one of the best ideas in the draft. It is also one of the easiest places for implementations to get subtly wrong. The first memo’s critique of the reference implementation is important precisely because it reveals that this property is stronger than a casual implementation will naturally deliver.

I recommend elevating this into conformance language. If an implementation cannot compute change-eligibility on the authorized projection, it should not claim projection-safe incremental sync. The protocol’s most privacy-distinctive property should not live as a soft expectation.

### 7. PDPP Should Be More Explicit About the Shape Limits of v0.1

The spec says personal data is represented as flat relational streams, with only `append_only` and `mutable_state`, and field allowlists are restricted to top-level field names in v0.1. Those choices are reasonable for a first version, but they should be presented more openly as shape constraints, not just neutral modeling choices.

This matters for two reasons. First, it explains why the first memo is right to call out verifiable credentials, CRDT/collaborative data, and high-frequency telemetry as poor fits. Second, it prevents readers from assuming that “personal data portability” has already been solved at a more general level than the current record model can honestly support.

### 8. Recommended Edits

I recommend the following near-term edits:

Add an explicit “enforcement status” distinction to the request and grant field tables. Mark stream/field/view/resource/time-range controls as protocol-enforced; mark retention as a policy commitment; mark most purpose semantics as advisory/auditable rather than enforced; mark `client_claims` as attributed and non-enforceable.

Add normative authorization-server requirements for rendering client-authored claims separately from protocol facts and manifest-derived data descriptions.

State prominently that revocation stops future access only, and add a separate deletion/erasure concept rather than implying revocation is sufficient.

Strengthen conformance around projection-safe incremental sync so that implementations cannot quietly degrade one of PDPP’s central privacy properties.

Move connector trust higher in the threat model and roadmap. The protocol already acknowledges the problem; it should treat it as foundational.

Adopt the first memo’s freshness and joint-data recommendations, but do so in a way that makes their enforcement status clear.

### Closing

PDPP’s central design move — separating manifests, grants, and disclosure from collection mechanics — is good. The protocol does not need a wholesale rethink. It needs sharper honesty about what is enforced where, and clearer treatment of the trust assumptions that sit outside the grant/query core. If the first memo is about making PDPP broader and more deployable, this memo is about making it more semantically disciplined. I think the two are complementary.
