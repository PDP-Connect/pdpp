Preamble

The General Data Portability Protocol (GDPP) is an open standard that defines how platforms and applications make personal data portable. GDPP specifies the supply side of data portability — the APIs, data formats, scope taxonomies, and authorization mechanisms that data-holding services implement so that users can collect, transfer, and control their own personal data.

Users have legal rights to their data under GDPR, the Digital Markets Act, CCPA, and emerging laws like the Utah Digital Choice Act. But no standard exists for how platforms should fulfill these rights in a structured, machine-readable way. Today, data portability depends on proprietary ZIP downloads, inconsistent formats, and — increasingly — scraping. GDPP replaces this fragmented landscape with a concrete, verifiable protocol that platforms can implement, regulators can reference, and the ecosystem can build on.

This document is a specification, not a proposal. It is designed to be implemented today. At the same time, GDPP is an evolving standard — we invite platforms, developers, policymakers, and civil society organizations to contribute to its development through the open governance process described in Part VI.

Part I: Why GDPP?

1. The Problem: Personal Data Trapped in Silos

Billions of people generate personal data every day across dozens of platforms — purchase histories on Amazon, health metrics on Fitbit, conversation logs on ChatGPT, watch patterns on YouTube, location data on Google Maps. This data is deeply personal, often sensitive, and almost always locked inside the platform that collected it.

The consequences are concrete:

A user preparing their taxes spends 2–4 hours manually downloading statements from multiple banks, categorizing hundreds of transactions, and cross-referencing Amazon purchase history — work that could take minutes if the data were portable.

A health-conscious user who wants to correlate sleep data from their Oura ring with activity data from their Apple Watch and dietary patterns from their purchase history simply cannot — the data lives in three separate silos with no common format or access mechanism.

A content creator analyzing their audience engagement across YouTube, Instagram, and TikTok must manually review thousands of data points across three separate dashboards, because there is no standard way to export and combine this data.

Today, the only mechanism for moving personal data between services is scraping — automated browser sessions that extract data from platform UIs using the user's credentials. Scraping is brittle (it breaks when UIs change), legally ambiguous (it operates in grey zones between TOS and user rights), and adversarial (platforms actively fight it). Yet scraping persists because no standard alternative exists.

GDPP replaces scraping with a protocol. It gives platforms a bounded, structured API for making user data portable. It gives users actual control over their data — not the theoretical control promised by regulations that lack technical implementation. And it gives the ecosystem a common standard to build on.

2. The Opportunity: From Scraping to Structured Portability

Data portability laws exist in every major jurisdiction. GDPR Article 20 guarantees the right to data portability. The Digital Markets Act requires continuous, real-time portability from gatekeepers. The California Consumer Privacy Act grants the right to access and transfer personal data. The Utah Digital Choice Act moves toward enforceable, machine-readable portability standards.

Yet in practice, data portability remains a fiction. Platforms offer ZIP file downloads in proprietary formats. Export processes are opaque, inconsistent, and designed for occasional human use — not for the continuous, programmatic access that applications and personal data infrastructure require.

GDPP transforms data portability from a legal right into a technical reality by defining:

What platforms must expose: A standard capability manifest describing available scopes and schemas.

How data is accessed: Standard scope-based APIs for structured data export, with user authorization via OAuth 2.0 consent.

What constitutes compliance: Measurable guarantees for response times, rate limits, and data freshness — preventing platforms from offering technically compliant but practically unusable portability.

How consent works: Verifiable authorization using OAuth 2.0 consent flows, with audit logging for compliance, ensuring users provably control who accesses their data and for what purpose.

The economic logic supports adoption. Platforms currently spend significant resources fighting unauthorized scraping. GDPP provides a controlled, auditable alternative that is better for platforms (predictable API load, no bot detection arms race), better for users (reliable data, clear consent), and better for the ecosystem (standard formats, interoperable tooling). The analogy to Open Banking is direct: banks resisted PSD2 for the same reasons (customer data was their competitive moat), but those that embraced it early gained trust and market share.

3. Design Philosophy

GDPP is built on six core principles:

1. Supply-Side Standardization. GDPP defines what platforms must implement. It standardizes data portability for data holders. The goal is a world where any platform that holds user data can expose it through a common, machine-readable interface.

2. User Sovereignty. The user controls when, what, and to whom their data is exported. Platforms cannot deny export requests for data the user owns. The protocol enforces this through verifiable consent and anti-discrimination requirements.

3. Capability-Based Discovery. Platforms publish machine-readable capability manifests at a well-known URL. Applications dynamically discover available data types without hard-coded integrations.

4. Scope-Driven Data Model. All data is organized into a hierarchical scope taxonomy ({source}.{category} — e.g., instagram.profile, youtube.watch_history). Data within each scope conforms to published JSON Schemas. This enables interoperability across platforms and downstream applications.

5. Verifiable Consent. Every data access request is backed by verifiable user consent through standard OAuth 2.0 authorization. Platforms verify consent before releasing data. This satisfies GDPR's "explicit consent" requirement and creates an auditable chain of authorization.

6. Ecosystem-Ready. GDPP does not prescribe what happens after data leaves the platform. Exported data flows into user-controlled infrastructure — personal servers, local applications, encrypted storage — where users grant downstream access to applications. GDPP focuses on the supply side; the demand side is governed by the user's own data infrastructure.

Part II: Protocol Overview

5. Architecture at a Glance

GDPP defines what Data Sources (platforms that hold user data) must implement so that user data can flow into Personal Data Infrastructure (user-controlled storage and access management).

┌──────────────────────────────────────────────────────────────┐ │ GDPP (This Specification) │ │ │ │ ┌─────────────┐ ┌─────────────┐ ┌─────────────┐ │ │ │ Instagram │ │ YouTube │ │ Gmail │ ... │ │ │ /.well-known│ │ /.well-known│ │ /.well-known│ │ │ │ /gdpp │ │ /gdpp │ │ /gdpp │ │ │ └──────┬──────┘ └──────┬──────┘ └──────┬──────┘ │ │ │ │ │ │ │ │ Scope-based export (user-authorized) │ │ │ JSON data conforming to GDPP schemas │ └─────────┼────────────────┼────────────────┼──────────────────┘ │ │ │ ▼ ▼ ▼ ┌──────────────────────────────────────────────────────────────┐ │ User's Personal Data Infrastructure │ │ │ │ Personal Servers, local applications, encrypted storage │ │ User grants access to downstream applications │ │ (Not specified by GDPP — governed by user's infrastructure) │ └──────────────────────────────────────────────────────────────┘ 

Each Data Source implements a GDPP endpoint — a set of standard APIs that authorized consumers use to discover available data, authenticate, and request exports. The protocol specifies:

Discovery: A well-known URL (/.well-known/gdpp) serving a machine-readable capability manifest.

Authentication: OAuth 2.0 authorization with GDPP-specific scopes.

Scope-based data access: Structured export of specific data scopes (e.g., instagram.profile, youtube.watch_history).

Transport: REST API (required), with optional MCP bindings.

6. Key Concepts

Data Source. A platform that holds user data and implements GDPP (e.g., Instagram, YouTube, Gmail, LinkedIn, Spotify). Also called "platform" or "provider." A Data Source is the supply side — it exposes data for portability.

User. The human whose data is being accessed. Identified by their account on the Data Source (e.g., email address, user ID). The user authorizes all data access.

Scope. A hierarchical identifier for a type of user data, using the naming convention {source}.{category} — for example, instagram.profile, youtube.watch_history, gmail.messages, amazon.orders. Scopes are the fundamental unit of data access in GDPP. Platforms declare which scopes they support in their capability manifest.

Schema. A JSON Schema definition for a scope. Schemas are versioned, immutable once published, and hosted at stable URLs. Core schemas cover common data scopes across platforms; extended schemas add platform-specific fields.

Capability Manifest. A JSON document at /.well-known/gdpp describing the Data Source's GDPP implementation — what scopes are available, how to authenticate, what schemas are used, and what rate limits apply.

Authorization Grant. An OAuth 2.0 authorization from a user permitting a consumer to access specific scopes. Grants are issued through the platform's OAuth authorization flow, include expiration and revocation semantics, and are recorded in the platform's audit log for verifiability.

Personal Data Infrastructure. The user-controlled systems that receive, store, and manage exported data — personal servers, desktop applications, encrypted cloud storage. GDPP does not specify these systems; it defines how data gets out of platforms so it can flow into whatever infrastructure the user chooses.

7. How Data Flows

GDPP enables two primary data flows:

Data Collection (Platform → User)

The user authorizes a data export from a platform. The platform verifies the authorization, assembles the requested scope data, and returns it in GDPP-standard JSON format.

Step

What Happens

1. Discovery

Consumer reads /.well-known/gdpp to learn what scopes are available

2. Authorization

User authorizes a grant specifying which scopes to export

3. Request

Consumer requests data for authorized scopes: GET /gdpp/v1/data/{scope}

4. Response

Platform returns structured JSON conforming to the scope's schema

5. Storage

Data flows into the user's personal data infrastructure

This replaces scraping. Instead of a browser automation script navigating Instagram's UI and extracting HTML, the consumer calls GET /gdpp/v1/data/instagram.profile and receives a structured JSON response.

Incremental Sync

For data that changes frequently, GDPP supports cursor-based incremental exports. A consumer can request only data that has changed since the last sync, using a cursor returned in the previous response. This enables continuous portability without repeated full exports.

8. Concrete Examples

Tax Preparation

A tax preparation application needs transaction data from multiple financial platforms. Without GDPP, the user spends 2–4 hours downloading statements from 3–5 banks, categorizing hundreds of transactions, and cross-referencing purchase history. With GDPP:

Tax app discovers scopes via /.well-known/gdpp on each financial platform.

User authorizes export of bank.transactions, creditcard.transactions, amazon.orders.

Tax app receives structured JSON for each scope — transactions already categorized, amounts normalized, dates standardized.

Tax app correlates transactions across sources, identifies deductions, and presents results.

Time: 2–3 minutes instead of 2–4 hours.

Health Pattern Analysis

A health insights application wants to correlate sleep, activity, and dietary patterns across wearable devices and purchase history:

App discovers scopes: oura.sleep, apple_health.activity, amazon.orders.

User authorizes export.

App receives structured data: sleep stages with timestamps, activity metrics, food purchase history.

App correlates: "You sleep 30% better on days with morning workouts."

Without GDPP: The user manually exports CSV files from three apps, reformats them into a spreadsheet, and attempts correlation by hand. Most users give up.

Content Creator Analytics

A content strategy tool analyzes a creator's consumption patterns across platforms:

App discovers scopes: youtube.watch_history, instagram.likes, tiktok.engagement.

User authorizes export.

App receives structured engagement data across platforms.

App identifies: "Your audience engages most with cooking tutorials posted Tuesday–Thursday mornings."

Without GDPP: The creator manually reviews thousands of videos across three platforms — a 2–3 hour process that most creators never do.

Part III: Normative Requirements

This section defines what "GDPP Compliant" means. These are enforceable requirements, not suggestions. The key words "MUST", "MUST NOT", "SHOULD", "SHOULD NOT", and "MAY" are to be interpreted as described in RFC 2119.

9. Portability Guarantees

9.1 The Problem

GDPR Article 20 requires portability "without hindrance." Without explicit, measurable guarantees, platforms can technically comply while making portability unusable — throttling exports to unusable speeds, marking requests as "async" indefinitely, or requiring "manual review" that never completes.

GDPP makes these behaviors protocol violations.

10. Freshness Guarantees

10.1 Why Freshness Matters

When data is exported, the consumer needs to know how current it is. A tax preparation app working with 6-month-old transaction data will miss recent purchases. A health app correlating sleep data needs data from the last few days, not last quarter. Freshness is the difference between useful data and misleading data.

11. Authorization and Consent

11.1 Authorization Method

Method

GDPP Mechanism

Legal Basis

OAuth 2.0

Standard OAuth 2.0 authorization code flow with GDPP-specific scopes, expiration, and grantee (client_id)

GDPR Art. 20 explicit consent

OAuth 2.0 is the standard authorization mechanism for GDPP. It leverages existing platform authentication infrastructure, is well-understood by developers, and provides verifiable consent through standard token validation.

11.2 Grant Structure

An authorization grant specifies:

User identity: The authenticated user on the Data Source platform

Grantee identity: The consumer's client_id (registered application or personal server)

Scopes: List of data scopes being authorized (e.g., gdpp:instagram.profile:read, gdpp:instagram.posts:read)

Expiration: Token lifetime after which the grant is invalid

Redirect URI: The consumer's registered callback URL

Grants are issued through the platform's OAuth 2.0 authorization flow and MUST be recorded in the platform's audit log for verifiability.

11.4 Revocation

When a user revokes a grant:

Before export starts: Data Source MUST cancel the export.

During export: Data Source SHOULD stop; MAY complete current batch.

After export complete: No effect on already-exported data.

Revocation affects future access. It does not recall data already exported — that data is now under the user's control. Revocations MUST be recorded in the platform's audit log. Platforms MUST support standard OAuth token revocation per RFC 7009.

11.5 Authorization UI Requirements

Data Sources MUST present clear authorization UI with:

List of data scopes being accessed, with human-readable labels

Identity of the requesting consumer

Clear "Approve" and "Deny" buttons

Link to revoke later

Data Sources MUST NOT include pre-checked sharing boxes, countdown timers that pressure approval, or bundled consent (export + marketing + AI training).

12. Data Ownership Classes

User data often involves other people. GDPP defines three ownership classes to balance portability with third-party privacy:

12.1 First-Party Data

Data created solely by or about the requesting user: profile information, user-authored posts, purchase history, watch history, settings and preferences.

Export rule: Always exportable at full fidelity.

12.2 Joint Data

Data involving the user and identifiable third parties: direct messages, shared documents, tagged photos, collaborative playlists.

Export rule: Exportable with transformations. The user's content is exported at full fidelity. Other parties' identifiable information is pseudonymized (e.g., user_abc123 instead of @john_smith). Other parties' PII is redacted or hashed.

12.3 Third-Party Data

Data primarily about others, observed by the user: other users' profiles, public posts the user has seen, detailed follower/following lists.

Export rule: Metadata only by default. Counts and aggregates are included. Identifiers are pseudonymized. Full content requires elevated consent and justification.

12.4 Manifest Declaration

Data Sources MUST declare data classes in their capability manifest:

{ "scope": "instagram.messages", "data_class": "joint", "transformations": ["pseudonymize_other_party", "redact_other_pii"] } 

13. Consumer Access Tiers

13.1 Two-Tier Model

GDPP mandates a two-tier access model to balance openness with security:

Tier 1: Baseline Access — available to any consumer with a valid user authorization grant, no registration required.

Export first-party data scopes

Minimum rate limit floors apply

Full historical and incremental sync

Tier 2: Registered Access — available to consumers that complete platform registration.

Access to joint and third-party data scopes

Higher rate limits

Webhook callbacks for async exports

Priority queue for large exports

13.2 Registration Rules

Data Sources MUST process registration within 48 hours. Data Sources MUST approve automatically if compliance requirements are met. Data Sources MUST provide specific denial reasons if rejecting. Data Sources MUST NOT require "business justification" for baseline scopes, deny based on competitive relationship, or charge fees for registration.

13.3 Baseline Access Scopes

The following scopes MUST be available without registration: *.profile, *.settings, *.posts (user-authored), *.purchases, *.watch_history, *.listen_history.

14. Non-Discrimination

Data portability access MUST be equal across all user tiers.

All users MUST have access to the same data scopes.

All users MUST receive full historical data.

Rate limit floors MUST apply equally to all users.

Export completion time limits MUST apply equally to all users.

Paid tiers MAY offer faster exports (above the floor), more frequent incremental syncs, and priority support. Paid tiers MUST NOT unlock additional data scopes, provide exclusive API features, or charge for exercising statutory rights.

15. Scope and Schema Governance

15.1 Scope Taxonomy

GDPP uses a hierarchical scope taxonomy: {source}.{category}. Examples:

Scope

Description

instagram.profile

Instagram profile information

instagram.posts

User-authored Instagram posts

youtube.watch_history

YouTube viewing history

amazon.orders

Amazon purchase history

oura.sleep

Oura ring sleep data

gmail.messages

Gmail messages

Source identifiers are registered in the scope registry. Category identifiers follow a common vocabulary where possible (profile, posts, messages, orders, watch_history, listen_history, transactions, settings).

15.2 Schema Registry Model

GDPP uses a federated schema registry with three tiers:

Core Schemas — canonical schemas for common scope categories (profiles, posts, messages, transactions). Versioned and immutable once published.

Extended Schemas — platform-specific fields beyond core schemas. MUST extend (not replace) core schemas. Consumers can safely ignore extended fields they don't understand.

Proprietary Schemas — truly platform-specific data with no common analog. No interoperability expectations. Clearly marked as proprietary.

15.3 Extension Rules

Extended schemas MUST include all core fields. Extended schemas MAY add platform-specific fields. Extended schemas MUST NOT redefine core field semantics.

15.4 Schema Licensing

All core and extended schemas are released under Apache 2.0 / CC0 dual license for maximum adoption. Schemas are pinned to content-addressable storage (e.g., IPFS) for permanence. Platforms MAY host schemas on their own domain.

16.1 Three Levels

Level 2: Silver — Full Portability All Bronze requirements, plus:

Export joint data with proper transformations

Provide user-accessible audit logging

Level 3: Gold — Ecosystem-Native Portability All Silver requirements, plus:

Real-time incremental sync (< 1 hour latency)

Third-party data with proper pseudonymization

Support MCP transport bindings

Webhook callbacks for all async operations

Schema extensions published to registry

Conformance test suite passing

16.2 Declaring Conformance

Data Sources declare their conformance level in the capability manifest:

{ "conformance": { "level": "silver", "certified_date": "2026-02-01", "test_report_url": "https://example.com/gdpp-test-report.json" } } 

Conformance is initially self-declared and verified through automated test suites. As the ecosystem matures, third-party auditors may offer formal certification.

19.2 OAuth 2.0 Authorization Flow

GDPP uses the standard OAuth 2.0 authorization code flow (RFC 6749) with GDPP-specific scopes. Consumers MUST use PKCE (RFC 7636) for all authorization requests.

Step 1: Authorization Request

The consumer redirects the user to the Data Source's authorization endpoint:

GET https://api.instagram.com/oauth/authorize? response_type=code& client_id=con_abc123& redirect_uri=https://server.example.com/callback& scope=gdpp:instagram.profile:read gdpp:instagram.posts:read& state=random_state_value& code_challenge=E9Melhoa2OwvFrEMTJguCHaoeK1t8URWbuGJSstw-cM& code_challenge_method=S256 

Step 2: User Consent

The Data Source presents the authorization UI (per Section 11.5), showing the requested scopes and consumer identity. The user approves or denies.

Step 3: Authorization Code Exchange

After user approval, the Data Source redirects to the consumer's callback with an authorization code:

OAuth scopes map directly to GDPP scopes using the format: gdpp:{source}.{category}:read (e.g., gdpp:instagram.profile:read, gdpp:instagram.posts:read).

Consumers authenticate to the Data Source API using the Authorization header with a Bearer token:

The Data Source validates the access token (via introspection per RFC 7662 or JWT verification), checks the token's scopes match the requested data, and returns the requested data.

The General Data Portability Protocol (GDPP) is an open standard that defines how platforms and applications make personal data portable. GDPP specifies the supply side of data portability — the APIs, data formats, scope taxonomies, and authorization mechanisms that data-holding services implement so that users can collect, transfer, and control their own personal data.

Users have legal rights to their data under GDPR, the Digital Markets Act, CCPA, and emerging laws like the Utah Digital Choice Act. Part I: Why GDPP?

1. The Problem: Personal Data Trapped in Silos

Billions of people generate personal data every day across dozens of platforms — purchase histories on Amazon, health metrics on Fitbit, conversation logs on ChatGPT, watch patterns on YouTube, location data on Google Maps. This data is deeply personal, often sensitive, and almost always locked inside the platform that collected it.

The consequences are concrete:

A user preparing their taxes spends 2–4 hours manually downloading statements from multiple banks, categorizing hundreds of transactions, and cross-referencing Amazon purchase history — work that could take minutes if the data were portable.

A health-conscious user who wants to correlate sleep data from their Oura ring with activity data from their Apple Watch and dietary patterns from their purchase history simply cannot — the data lives in three separate silos with no common format or access mechanism.

A content creator analyzing their audience engagement across YouTube, Instagram, and TikTok must manually review thousands of data points across three separate dashboards, because there is no standard way to export and combine this data.

Today, the only mechanism for moving personal data between services is scraping — automated browser sessions that extract data from platform UIs using the user's credentials. Scraping is brittle (it breaks when UIs change), legally ambiguous (it operates in grey zones between TOS and user rights), and adversarial (platforms actively fight it). Yet scraping persists because no standard alternative exists.

GDPP replaces scraping with a protocol. It gives platforms a bounded, structured API for making user data portable. It gives users actual control over their data — not the theoretical control promised by regulations that lack technical implementation. And it gives the ecosystem a common standard to build on.

2. The Opportunity: From Scraping to Structured Portability

Data portability laws exist in every major jurisdiction. GDPR Article 20 guarantees the right to data portability. The Digital Markets Act requires continuous, real-time portability from gatekeepers. The California Consumer Privacy Act grants the right to access and transfer personal data. The Utah Digital Choice Act moves toward enforceable, machine-readable portability standards.

Yet in practice, data portability remains a fiction. Platforms offer ZIP file downloads in proprietary formats. Export processes are opaque, inconsistent, and designed for occasional human use — not for the continuous, programmatic access that applications and personal data infrastructure require.

GDPP transforms data portability from a legal right into a technical reality by defining:

What platforms must expose: A standard capability manifest describing available scopes and schemas.

How data is accessed: Standard scope-based APIs for structured data export, with user authorization via OAuth 2.0 consent.

What constitutes compliance: Measurable guarantees for response times, rate limits, and data freshness — preventing platforms from offering technically compliant but practically unusable portability.

How consent works: Verifiable authorization using OAuth 2.0 consent flows, with audit logging for compliance, ensuring users provably control who accesses their data and for what purpose.

The economic logic supports adoption. Platforms currently spend significant resources fighting unauthorized scraping. GDPP provides a controlled, auditable alternative that is better for platforms (predictable API load, no bot detection arms race), better for users (reliable data, clear consent), and better for the ecosystem (standard formats, interoperable tooling). The analogy to Open Banking is direct: banks resisted PSD2 for the same reasons (customer data was their competitive moat), but those that embraced it early gained trust and market share.

3. Design Philosophy

GDPP is built on six core principles:

1. Supply-Side Standardization. GDPP defines what platforms must implement. It standardizes data portability for data holders. The goal is a world where any platform that holds user data can expose it through a common, machine-readable interface.

2. User Sovereignty. The user controls when, what, and to whom their data is exported. Platforms cannot deny export requests for data the user owns. The protocol enforces this through verifiable consent and anti-discrimination requirements.

3. Capability-Based Discovery. Platforms publish machine-readable capability manifests at a well-known URL. Applications dynamically discover available data types without hard-coded integrations.

4. Scope-Driven Data Model. All data is organized into a hierarchical scope taxonomy ({source}.{category} — e.g., instagram.profile, youtube.watch_history). Data within each scope conforms to published JSON Schemas. This enables interoperability across platforms and downstream applications.

5. Verifiable Consent. Every data access request is backed by verifiable user consent through standard OAuth 2.0 authorization. Platforms verify consent before releasing data. This satisfies GDPR's "explicit consent" requirement and creates an auditable chain of authorization.

6. Ecosystem-Ready. GDPP does not prescribe what happens after data leaves the platform. Exported data flows into user-controlled infrastructure — personal servers, local applications, encrypted storage — where users grant downstream access to applications. GDPP focuses on the supply side; the demand side is governed by the user's own data infrastructure.

Part II: Protocol Overview

5. Architecture at a Glance

GDPP defines what Data Sources (platforms that hold user data) must implement so that user data can flow into Personal Data Infrastructure (user-controlled storage and access management).

┌──────────────────────────────────────────────────────────────┐ │ GDPP (This Specification) │ │ │ │ ┌─────────────┐ ┌─────────────┐ ┌─────────────┐ │ │ │ Instagram │ │ YouTube │ │ Gmail │ ... │ │ │ /.well-known│ │ /.well-known│ │ /.well-known│ │ │ │ /gdpp │ │ /gdpp │ │ /gdpp │ │ │ └──────┬──────┘ └──────┬──────┘ └──────┬──────┘ │ │ │ │ │ │ │ │ Scope-based export (user-authorized) │ │ │ JSON data conforming to GDPP schemas │ └─────────┼────────────────┼────────────────┼──────────────────┘ │ │ │ ▼ ▼ ▼ ┌──────────────────────────────────────────────────────────────┐ │ User's Personal Data Infrastructure │ │ │ │ Personal Servers, local applications, encrypted storage │ │ User grants access to downstream applications │ │ (Not specified by GDPP — governed by user's infrastructure) │ └──────────────────────────────────────────────────────────────┘ 

Each Data Source implements a GDPP endpoint — a set of standard APIs that authorized consumers use to discover available data, authenticate, and request exports. The protocol specifies:

Discovery: A well-known URL (/.well-known/gdpp) serving a machine-readable capability manifest.

Authentication: OAuth 2.0 authorization with GDPP-specific scopes.

Scope-based data access: Structured export of specific data scopes (e.g., instagram.profile, youtube.watch_history).

Transport: REST API (required), with optional MCP bindings.

6. Key Concepts

Data Source. A platform that holds user data and implements GDPP (e.g., Instagram, YouTube, Gmail, LinkedIn, Spotify). Also called "platform" or "provider." A Data Source is the supply side — it exposes data for portability.

User. The human whose data is being accessed. Identified by their account on the Data Source (e.g., email address, user ID). The user authorizes all data access.

Scope. A hierarchical identifier for a type of user data, using the naming convention {source}.{category} — for example, instagram.profile, youtube.watch_history, gmail.messages, amazon.orders. Scopes are the fundamental unit of data access in GDPP. Platforms declare which scopes they support in their capability manifest.

Schema. A JSON Schema definition for a scope. Schemas are versioned, immutable once published, and hosted at stable URLs. Core schemas cover common data scopes across platforms; extended schemas add platform-specific fields.

Capability Manifest. A JSON document at /.well-known/gdpp describing the Data Source's GDPP implementation — what scopes are available, how to authenticate, what schemas are used, and what rate limits apply.

Authorization Grant. An OAuth 2.0 authorization from a user permitting a consumer to access specific scopes. Grants are issued through the platform's OAuth authorization flow, include expiration and revocation semantics, and are recorded in the platform's audit log for verifiability.

Personal Data Infrastructure. The user-controlled systems that receive, store, and manage exported data — personal servers, desktop applications, encrypted cloud storage. GDPP does not specify these systems; it defines how data gets out of platforms so it can flow into whatever infrastructure the user chooses.

7. How Data Flows

GDPP enables two primary data flows:

Data Collection (Platform → User)

The user authorizes a data export from a platform. The platform verifies the authorization, assembles the requested scope data, and returns it in GDPP-standard JSON format.

Step

What Happens

1. Discovery

Consumer reads /.well-known/gdpp to learn what scopes are available

2. Authorization

User authorizes a grant specifying which scopes to export

3. Request

Consumer requests data for authorized scopes: GET /gdpp/v1/data/{scope}

4. Response

Platform returns structured JSON conforming to the scope's schema

5. Storage

Data flows into the user's personal data infrastructure

This replaces scraping. Instead of a browser automation script navigating Instagram's UI and extracting HTML, the consumer calls GET /gdpp/v1/data/instagram.profile and receives a structured JSON response.

Incremental Sync

For data that changes frequently, GDPP supports cursor-based incremental exports. A consumer can request only data that has changed since the last sync, using a cursor returned in the previous response. This enables continuous portability without repeated full exports.

Part III: Normative Requirements

This section defines what "GDPP Compliant" means. These are enforceable requirements, not suggestions. The key words "MUST", "MUST NOT", "SHOULD", "SHOULD NOT", and "MAY" are to be interpreted as described in RFC 2119.

GDPR Article 20 requires portability "without hindrance." GDPP makes these behaviors protocol violations.

10. Freshness Guarantees

10.1 Why Freshness Matters

When data is exported, the consumer needs to know how current it is. A tax preparation app working with 6-month-old transaction data will miss recent purchases. A health app correlating sleep data needs data from the last few days, not last quarter. Freshness is the difference between useful data and misleading data.

11.1 Authorization Method

Method

OAuth 2.0 is the standard authorization mechanism for GDPP. It leverages existing platform authentication infrastructure, is well-understood by developers, and provides verifiable consent through standard token validation.

11.2 Grant Structure

An authorization grant specifies:

User identity: The authenticated user on the Data Source platform

Grantee identity: The consumer's client_id (registered application or personal server)

Scopes: List of data scopes being authorized (e.g., gdpp:instagram.profile:read, gdpp:instagram.posts:read)

Expiration: Token lifetime after which the grant is invalid

Redirect URI: The consumer's registered callback URL

Grants are issued through the platform's OAuth 2.0 authorization flow and MUST be recorded in the platform's audit log for verifiability.

11.4 Revocation

When a user revokes a grant:

Before export starts: Data Source MUST cancel the export.

During export: Data Source SHOULD stop; MAY complete current batch.

After export complete: No effect on already-exported data.

Revocation affects future access. It does not recall data already exported — that data is now under the user's control. Revocations MUST be recorded in the platform's audit log. Platforms MUST support standard OAuth token revocation per RFC 7009.

11.5 Authorization UI Requirements

Data Sources MUST present clear authorization UI with:

List of data scopes being accessed, with human-readable labels

Identity of the requesting consumer

Clear "Approve" and "Deny" buttons

Link to revoke later

Data Sources MUST NOT include pre-checked sharing boxes, countdown timers that pressure approval, or bundled consent (export + marketing + AI training).

12. Data Ownership Classes

User data often involves other people. GDPP defines three ownership classes to balance portability with third-party privacy:

12.1 First-Party Data

Data created solely by or about the requesting user: profile information, user-authored posts, purchase history, watch history, settings and preferences.

Export rule: Always exportable at full fidelity.

12.2 Joint Data

Data involving the user and identifiable third parties: direct messages, shared documents, tagged photos, collaborative playlists.

Export rule: Exportable with transformations. The user's content is exported at full fidelity. Other parties' identifiable information is pseudonymized (e.g., user_abc123 instead of @john_smith). Other parties' PII is redacted or hashed.

12.3 Third-Party Data

Data primarily about others, observed by the user: other users' profiles, public posts the user has seen, detailed follower/following lists.

Export rule: Metadata only by default. Counts and aggregates are included. Identifiers are pseudonymized. Full content requires elevated consent and justification.

12.4 Manifest Declaration

Data Sources MUST declare data classes in their capability manifest:

{ "scope": "instagram.messages", "data_class": "joint", "transformations": ["pseudonymize_other_party", "redact_other_pii"] } 

13. Consumer Access Tiers

13.1 Two-Tier Model

GDPP mandates a two-tier access model to balance openness with security:

Tier 1: Baseline Access — available to any consumer with a valid user authorization grant, no registration required.

Export first-party data scopes

Minimum rate limit floors apply

Full historical and incremental sync

Tier 2: Registered Access — available to consumers that complete platform registration.

Access to joint and third-party data scopes

Higher rate limits

Webhook callbacks for async exports

Priority queue for large exports

13.2 Registration Rules

Data Sources MUST process registration within 48 hours. Data Sources MUST approve automatically if compliance requirements are met. Data Sources MUST provide specific denial reasons if rejecting. Data Sources MUST NOT require "business justification" for baseline scopes, deny based on competitive relationship, or charge fees for registration.

13.3 Baseline Access Scopes

The following scopes MUST be available without registration: *.profile, *.settings, *.posts (user-authored), *.purchases, *.watch_history, *.listen_history.

14. Non-Discrimination

Data portability access MUST be equal across all user tiers.

All users MUST have access to the same data scopes.

All users MUST receive full historical data.

Rate limit floors MUST apply equally to all users.

Export completion time limits MUST apply equally to all users.

Paid tiers MAY offer faster exports (above the floor), more frequent incremental syncs, and priority support. Paid tiers MUST NOT unlock additional data scopes, provide exclusive API features, or charge for exercising statutory rights.

15.1 Scope Taxonomy

Source identifiers are registered in the scope registry. Category identifiers follow a common vocabulary where possible (profile, posts, messages, orders, watch_history, listen_history, transactions, settings).

15.2 Schema Registry Model

GDPP uses a federated schema registry with three tiers:

Core Schemas — canonical schemas for common scope categories (profiles, posts, messages, transactions). Versioned and immutable once published.

Extended Schemas — platform-specific fields beyond core schemas. MUST extend (not replace) core schemas. Consumers can safely ignore extended fields they don't understand.

Proprietary Schemas — truly platform-specific data with no common analog. No interoperability expectations. Clearly marked as proprietary.

15.3 Extension Rules

Extended schemas MUST include all core fields. Extended schemas MAY add platform-specific fields. Extended schemas MUST NOT redefine core field semantics.

15.4 Schema Licensing

All core and extended schemas are released under Apache 2.0 / CC0 dual license for maximum adoption. Schemas are pinned to content-addressable storage (e.g., IPFS) for permanence. Platforms MAY host schemas on their own domain.

19.1 Authentication Mode

19.2 OAuth 2.0 Authorization Flow

GDPP uses the standard OAuth 2.0 authorization code flow (RFC 6749) with GDPP-specific scopes. Consumers MUST use PKCE (RFC 7636) for all authorization requests.

Step 1: Authorization Request

The consumer redirects the user to the Data Source's authorization endpoint:

GET https://api.instagram.com/oauth/authorize? response_type=code& client_id=con_abc123& redirect_uri=https://server.example.com/callback& scope=gdpp:instagram.profile:read gdpp:instagram.posts:read& state=random_state_value& code_challenge=E9Melhoa2OwvFrEMTJguCHaoeK1t8URWbuGJSstw-cM& code_challenge_method=S256 

Step 2: User Consent

The Data Source presents the authorization UI (per Section 11.5), showing the requested scopes and consumer identity. The user approves or denies.

Step 3: Authorization Code Exchange

After user approval, the Data Source redirects to the consumer's callback with an authorization code:

POST https://api.instagram.com/oauth/access_token Content-Type: application/x-www-form-urlencoded grant_type=authorization_code& code=auth_code_xyz& client_id=con_abc123& client_secret=secret_xyz& redirect_uri=https://server.example.com/callback& code_verifier=dBjftJeZ4CVP-mB92K27uhbUJU1p1r_wW1gFWFOEjXk 

Step 4: Access Token Response

{ "access_token": "gdpp_at_abc123xyz", "token_type": "Bearer", "expires_in": 3600, "refresh_token": "gdpp_rt_def456uvw", "scope": "gdpp:instagram.profile:read gdpp:instagram.posts:read" } 

OAuth scopes map directly to GDPP scopes using the format: gdpp:{source}.{category}:read (e.g., gdpp:instagram.profile:read, gdpp:instagram.posts:read).

19.3 Consumer Authentication

Consumers authenticate to the Data Source API using the Authorization header with a Bearer token:

Authorization: Bearer <access_token> 

The Data Source validates the access token (via introspection per RFC 7662 or JWT verification), checks the token's scopes match the requested data, and returns the requested data.

Part V: Interoperability

27. Relationship to Public Data Protocols

GDPP focuses on personal, private data — information that belongs to individual users and requires their explicit consent to access. This distinguishes GDPP from protocols that handle public or semi-public data:

27.1 DSNP (Decentralized Social Networking Protocol)

DSNP defines a shared social layer for public social interactions — posts, follows, reactions — that are designed to be public by nature. GDPP and DSNP are complementary:

Aspect

DSNP

GDPP

Data type

Public social graph, public posts

Private personal data

Access model

Public by default

Consent-required

Primary consumer

Social applications

Personal data infrastructure, applications

Portability model

User carries public identity across apps

User exports private data from platforms

Interoperability: A user's complete digital identity spans both protocols. Their public posts and social graph live on DSNP. Their private messages, purchase history, and personal preferences are accessible via GDPP. Applications that serve users well need access to both layers.

27.2 AT Protocol (Bluesky)

The AT Protocol enables federated social networking with user-controlled identity and portable public data. Like DSNP, it handles public social interactions. GDPP handles the private complement:

Aspect

AT Protocol

GDPP

Data type

Public social posts, follows

Private personal data

Architecture

Federated servers, user-hosted repos

Platform-implemented APIs

Identity

Decentralized identifiers (DIDs)

Platform accounts (OAuth 2.0)

Portability

User moves their repo between servers

User exports data from platforms

Interoperability: AT Protocol handles where a user's public social presence lives. GDPP handles how their private data (from any platform) is made portable. GDPP uses standard OAuth 2.0, which can integrate with AT Protocol's authentication system where platforms support both protocols.

27.3 The Complete Picture

GDPP schemas are designed to coexist with these protocols. The scope namespace ({source}.{category}) avoids collision with DSNP or AT Protocol identifiers. Applications that access multiple protocols can combine public context (DSNP/AT) with private data (GDPP) to serve users comprehensively.

28. Integration with Personal Data Infrastructure

GDPP defines how data gets out of platforms. What happens after export is governed by the user's chosen personal data infrastructure — whether that is a personal server, a local application, or a cloud-hosted data vault.

Call for Collaboration

GDPP is an open standard that benefits from broad participation. We invite:

Platforms to implement GDPP and shape the schemas for their data scopes. Early adopters define what their data looks like in the standard.

Developers to build applications and personal data infrastructure that consume GDPP APIs, providing feedback on what works and what needs improvement.

Policymakers to reference GDPP as a technical benchmark for data portability compliance.

Civil society organizations to participate in schema governance and advocate for user-centered design.

Researchers to study GDPP's impact on data portability, competition, and user agency.

Contributions are welcome through the open specification repository. All contributions are governed by the project's code of conduct and contribution guidelines.
