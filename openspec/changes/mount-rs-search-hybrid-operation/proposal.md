## Why

The `define-reference-operation-environments` proof sequence has landed `rs.streams.list`, `rs.streams.detail`, `rs.schema.get`, `rs.records.list`, `rs.records.get`, `rs.search.lexical`, and `rs.search.semantic` as canonical operation capsules mounted from the native Fastify host (and, where applicable, the Next sandbox host). The next remaining public AS/RS surface that still embeds public-contract semantics directly in a transport-shaped helper is `GET /v1/search/hybrid`, served by `runHybridSearch` in `reference-implementation/server/search-hybrid.js`. The route handler is already thin, but `runHybridSearch` mixes the public-contract slice (allowlist + explicit `cursor` rejection + forbidden parameters, `q`-required, `limit` clamping, `streams[]` normalization, `filter[...]` coupling, sub-request fan-out, round-robin merge with dedup by `(connector_id, stream, record_key)`, per-source score forwarding under `scores`, `retrieval_mode: "hybrid"` and `retrieval_sources` provenance, list-envelope shape, and `disclosure.served` data block) with native helper-import wiring for `runLexicalSearch` and `runSemanticSearch`.

Hybrid retrieval has properties unique to it: it is the only retrieval surface that does NOT support cursor pagination in v1 (the spec's explicit pagination choice — see `define-hybrid-retrieval/specs/hybrid-retrieval/spec.md`); it is a composition over two independently-grant-enforced retrieval surfaces and SHALL NOT introduce a new grant-logic path; per-hit scores are advertised per-source under `scores` rather than on a flat `score` field because the underlying surfaces use different score kinds, orders, and value semantics that v1 does NOT normalize. This change extracts only the host-independent slice into an operation capsule and keeps the lexical and semantic runners (and therefore all grant enforcement, plan compilation, snapshot orchestration, ranking, and snippet hydration) as capability dependencies; it does not change hybrid scoring semantics, does not introduce hybrid cursors, does not normalize lexical and semantic score values together, and does not rewrite `server/search-hybrid.js`.

## What Changes

- Introduce a canonical `rs.search.hybrid` operation implementation that owns the host-independent slice of public hybrid search behavior: strict v1 query-param allowlist (`q`, `limit`, `streams`, `streams[]`, `filter`), explicit `cursor` rejection (raised as `invalid_request` with `param: 'cursor'` — v1 hybrid does NOT support cursor pagination), the explicit forbidden-parameter list (`vector`, `embedding`, `embed`, `model`, `model_id`, `model_family`, `rank`, `boost`, `weights`, `blend`, `connector_id`, `fields`, `expand`, `expand[]`, `expand_limit`, `expand_limit[]`, `order`, `sort`, `mode`), `q`-required check, `limit` clamping, `streams[]` normalization, `filter[...]` coupling, mode classification, per-source fan-out via lexical and semantic runner dependencies (which retain advertisement, grant projection, stream-grant intersection, field-grant intersection, and record-level grant enforcement), round-robin merge with dedup by `(connector_id, stream, record_key)`, `matched_fields` union (lexical-first), per-source score forwarding under a `scores` map (verbatim, no normalization), first-non-empty snippet preservation, `retrieval_sources` provenance (subset of `["lexical", "semantic"]`, lexical-first order), `retrieval_mode: "hybrid"`, list-envelope shape (no `next_cursor` in v1), and `disclosure.served` data block (`query_shape: 'search_hybrid'`, `record_count`, `has_more`, `mode`, `lexical_count`, `semantic_count`).
- Mount the operation from the native Fastify reference server (`GET /v1/search/hybrid`), preserving response shape, error codes (including `cursor` rejection and the explicit forbidden-parameter list), per-source `scores` map shape, `retrieval_sources` provenance, `retrieval_mode: "hybrid"`, dedup semantics, grant filtering through the underlying runners, advertisement gating (the route remains registered only when both lexical and semantic retrieval are advertised), and disclosure-spine semantics.
- Reduce native `runHybridSearch` in `reference-implementation/server/search-hybrid.js` to a thin native dependency-wiring shell around `executeSearchHybrid`. The shell builds the synthetic sub-request used by `runLexicalSearch` and `runSemanticSearch`, and translates the operation's typed `SearchHybridRequestError` back into the previous plain-`Error` shape (`err.code`, optional `err.param`) so the route's existing `rejectQuery` / `handleError` paths continue to work. Keep `parseHybridSearchParams` exported as a delegating shim for any direct importers.
- Native `server/search-hybrid.js` keeps the lexical and semantic runner imports (`runLexicalSearch`, `runSemanticSearch`); the operation receives them as capability dependencies (`runLexical`, `runSemantic`) and consumes only the per-source result envelopes — it does NOT itself import either underlying retrieval surface.
- Extend the boundary tests so the new operation module is gated by the shared `operation-boundary.js` rule and add a per-operation boundary test asserting the operation does not statically import `server/search.js`, `server/search-semantic.js`, or `server/search-hybrid.js` (so the operation cannot become a back door into either underlying retrieval surface or its own native shell).
- Do not change lexical or semantic operation contracts except for tiny test fixtures/imports needed by hybrid (none required).
- Do not implement hybrid pagination. Do not change hybrid scoring semantics. Do not normalize lexical and semantic score values together. Do not add a sandbox `/sandbox/v1/search/hybrid` route — the sandbox today advertises lexical retrieval only and does not register a semantic surface, so a sandbox hybrid surface would require both a sandbox semantic fixture and a truthful hybrid advertisement, which is out of scope.

## Capabilities

### New Capabilities

- None.

### Modified Capabilities

- `reference-implementation-architecture`: `rs.search.hybrid` becomes operation-owned, joining the existing canonical operations.

## Impact

- Affected code: `reference-implementation/operations/rs-search-hybrid/**`, `reference-implementation/server/index.js` (`GET /v1/search/hybrid` route only — wiring change, no contract change), `reference-implementation/server/search-hybrid.js` (the `runHybridSearch` shell becomes a thin native adapter; lexical/semantic runner imports stay), `reference-implementation/package.json` (operation export), and tests.
- No public API shape change for `GET /v1/search/hybrid`: it continues to return its existing JSON envelope (`object: 'list'`, `url: '/v1/search/hybrid'`, `has_more`, `data: search_result[]` — no `next_cursor` in v1), error codes, the explicit `cursor` rejection (any cursor on the wire ⇒ `invalid_request` with `param: 'cursor'`), the explicit forbidden-parameter list, per-hit `retrieval_mode: "hybrid"`, per-hit `retrieval_sources` provenance (subset of `["lexical", "semantic"]`, lexical-first order), per-source `scores` map (each entry is the underlying surface's score object verbatim — no normalization, no flat `score` field), grant filtering behavior (delegated to the underlying runners), stream/filter query semantics, and disclosure-spine event shape (`query_shape: 'search_hybrid'`).
- Grant enforcement remains delegated to the underlying lexical and semantic runners. The operation does NOT introduce a new grant-logic path. Errors from either runner (e.g. `grant_stream_not_allowed`) propagate unchanged — hybrid behaves identically to calling the underlying endpoints for the same grant.
- v1 hybrid pagination remains unsupported. The advertisement at `/.well-known/oauth-protected-resource` continues to advertise `cursor_supported: false` for the hybrid extension. Clients that need paging beyond `limit` should fall back to the individual `/v1/search` and `/v1/search/semantic` endpoints.
- No sandbox hybrid route is added. The sandbox's advertised capability set continues to declare lexical retrieval only.
