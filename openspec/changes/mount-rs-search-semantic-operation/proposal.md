## Why

The `define-reference-operation-environments` proof sequence has landed `rs.streams.list`, `rs.streams.detail`, `rs.schema.get`, `rs.records.list`, `rs.records.get`, and `rs.search.lexical` as canonical operation capsules mounted from the native Fastify host (and, where applicable, the Next sandbox host). The next remaining public AS/RS surface that still embeds public-contract semantics directly in a transport-shaped helper is `GET /v1/search/semantic`, served by `runSemanticSearch` in `reference-implementation/server/search-semantic.js`. The route handler is already thin, but `runSemanticSearch` mixes the public-contract slice (allowlist + forbidden parameters, `q`-required, `limit` clamping, `streams[]` normalization, `filter[...]` coupling, cross-stream advertisement gate, mode classification, cursor encode/decode with the `sem1.` prefix, snapshot orchestration with backend-identity stale-cursor detection, slice math, score-advertisement gate, `search_result` shaping including `retrieval_mode: "semantic"`, list-envelope shape, and `disclosure.served` data block) with embedding-backend, vector-index, and records-table snippet hydration concerns.

Semantic retrieval is a higher-risk surface than lexical: backend identity (profile/model/dimensions/distance metric) is part of the capability advertisement and the cursor's stale-cursor invariant; recall determinism, snippet grant safety, and the no-silent-fallback property are architectural requirements; and the `retrieval_mode: "semantic"` and `score.kind: "semantic_distance"` semantics must not collapse into a generic search shape. This change extracts only the host-independent slice into an operation capsule and keeps the embedding backend, vector index, ranking, snapshot persistence, and snippet hydration behind capability dependencies; it does not extract a production `SemanticIndex` interface, does not introduce Postgres or pgvector, does not normalize lexical and semantic score semantics together, and does not rewrite `server/search-semantic.js`.

## What Changes

- Introduce a canonical `rs.search.semantic` operation implementation that owns the host-independent slice of public semantic search behavior: strict v1 query-param allowlist with the explicit forbidden-parameter list (`vector`, `embedding`, `embed`, `model`, `model_id`, `model_family`, `rank`, `boost`, `weights`, `blend`, `connector_id`, `fields`, `expand`, `expand[]`, `expand_limit`, `expand_limit[]`, `order`, `sort`, `mode`), `q`-required check, `limit` clamping, `streams[]` normalization, `filter[...]` coupling, cross-stream advertisement gate, mode classification, client-mode `streams[] ⊆ grant.streams` rejection, owner-mode soft `streams[]` filter, cursor encode/decode with the `sem1.` prefix, snapshot orchestration including backend-identity stale-cursor detection, slice math, score-advertisement gate (gated on `score.kind: "semantic_distance"` lower-is-better), `search_result` shaping (including `retrieval_mode: "semantic"` and grant-safe verbatim snippet emission delegated to a dependency), list-envelope shape, and `disclosure.served` data block.
- Mount the operation from the native Fastify reference server (`GET /v1/search/semantic`), preserving response shape, error codes, cursor semantics, scoring semantics, grant filtering, backend/profile/model identity disclosure, and disclosure-spine semantics.
- Reduce native `runSemanticSearch` in `reference-implementation/server/search-semantic.js` to a thin native dependency-wiring shell around `executeSearchSemantic`. Keep `parseSemanticSearchParams` exported as a delegating shim so existing direct importers (e.g. `semantic-retrieval.test.js`) continue to receive the same error shape.
- Native `server/search-semantic.js` keeps the embedding backend factories, vector-index implementations (sqlite-vec and blob-flat), backfill, drift detection, snippet hydration from the records table, snapshot persistence, and the no-fallback invariant (no import from `server/search.js`); the operation receives them as capability dependencies (`getAdvertisement`, `getCurrentBackendIdentity`, `listOwnerVisibleConnectorIds`, `resolveOwnerManifestForConnector`, `buildOwnerReadGrantForManifest`, `resolveClientManifest`, `buildSearchPlanForGrant`, `buildSnapshot`, `persistSnapshot`, `loadSnapshot`, `hydrateResult`, `formatRecordUrl`).
- Extend the boundary tests so the new operation module is gated by the shared `operation-boundary.js` rule and add a per-operation boundary test asserting the operation does not statically import `server/search-semantic.js` or `server/search.js` (the no-silent-fallback invariant continues to apply at the operation boundary).
- Do not migrate hybrid retrieval, attachment blobs, runs, traces, or any `_ref` route in this slice.
- Do not add a sandbox `/sandbox/v1/search/semantic` route. The current sandbox advertises lexical search only and does not claim semantic retrieval; introducing a sandbox semantic route would require a truthful mock-backed semantic fixture with explicit capability metadata, which is out of scope for this slice.

## Capabilities

### New Capabilities

- None.

### Modified Capabilities

- `reference-implementation-architecture`: `rs.search.semantic` becomes operation-owned, joining the existing canonical operations.

## Impact

- Affected code: `reference-implementation/operations/rs-search-semantic/**`, `reference-implementation/server/index.js` (`GET /v1/search/semantic` route only — wiring change, no contract change), `reference-implementation/server/search-semantic.js` (the `runSemanticSearch` shell becomes a thin native adapter; embedding backends, vector indexes, backfill/drift, and snippet hydration stay), `reference-implementation/package.json` (operation export), and tests.
- No public API shape change for `GET /v1/search/semantic`: it continues to return its existing JSON envelope (`object: 'list'`, `url: '/v1/search/semantic'`, `has_more`, optional `next_cursor`, `data: search_result[]`), error codes, cursor format (`sem1.<base64url-json>`), per-hit score shape (exactly `{ kind: "semantic_distance", value, order: "lower_is_better" }` — no capability-level metadata fields are repeated on individual hits), `retrieval_mode: "semantic"` per hit, grant filtering behavior, stream/filter query semantics, and disclosure-spine event shape. Backend/profile/model identity disclosure stays on the capability surface (`capabilities.semantic_retrieval.score` at `/.well-known/oauth-protected-resource` continues to advertise `value_semantics`, `comparable_with`, model, dimensions, distance_metric, and where applicable profile_id/dtype); the operation does not synthesize or reshape that advertisement. The existing forbidden-parameter list and the cross-surface cursor rejection (`sem1.` prefix gating) are preserved.
- No production `SemanticIndex` interface is extracted, no pgvector or Postgres adapter is introduced, no semantic score normalization with lexical is performed, and no broad rewrite of `server/search-semantic.js` is performed. Backend identity remains a public capability surface — the operation advertises score comparability and stale-cursor identity through dependency-supplied metadata, and the native shell continues to hold the embedding and vector-index implementations.
- No sandbox semantic route is added. The sandbox's advertised capability set continues to declare lexical retrieval only.
