# Design: Source-Backed Fulfillment

## Context

The reference implementation proves a collected-data model: connectors populate local canonical records; the resource server enforces grants over them. Core never requires that model — `spec-core.md` §1 states the protocol "is useful without" collection, and `spec-architecture.md` states "fulfillment remains an implementation choice: serve from storage, collect on a schedule, or collect before serving." This change names and bounds the fourth choice that sentence leaves open: satisfy the read by fetching from the upstream source at request time, with grant enforcement unchanged and local.

Naming: the feature is **source-backed fulfillment**. "Passthrough," the working name in the 2026-06-04 design note, is rejected for the durable vocabulary: it collides with the MCP "token passthrough" security anti-pattern (an MCP server forwarding tokens not issued to it — this design does the opposite and says so normatively), with ~30 unrelated code-level usages in this repo, and with the distinct "bulk-import/passthrough drain" idea in `design-notes/gap-backlog-drain-depth-2026-06-09.md`. "Federation" is rejected because it implies cross-source joins this design does not perform. The per-response provenance vocabulary is `origin: live | cache` (absence = retained), matching the design note's live/cache/retained triad.

## The fulfillment spectrum (and what clients see of it)

Implementation postures form a spectrum: retained canonical records → imported bootstrap (bulk-import note) → source-backed with cache → source-backed with local secondary indexes → fully live. The central decision: **the spectrum is not a client-facing enum.** Clients see exactly four things, all of which already exist or extend existing surfaces:

1. **Capability** — the effective per-stream `query` declaration and schema `field_capabilities` say what request shapes work (the stream-level surface when retained, the declared source-backed subset when live); undeclared shapes are HTTP 400 (existing normative rule: "unsupported query shapes MUST be rejected... and MUST NOT be silently ignored").
2. **Freshness** — the Core `freshness` object, with `captured_at` = upstream fetch time. Live reads make freshness *better*, not different in kind.
3. **Provenance** — `meta.fulfillment.origin` disclosing live vs cache on source-backed responses.
4. **Failure** — `source_unavailable` (503) when the upstream cannot answer and no honest cache can.

This keeps the promise surface small and reuses the honesty vocabulary the repo already ratified (per-stream `query` declarations, `freshness`, maturity/coverage states), per the explicit instruction in the passthrough note to reuse rather than invent.

## Which PDPP semantics are mandatory regardless of fulfillment mode

From the spec map (`tmp/workstreams/sbf-spec-map-2026-07-09.md`, verified against `spec-core.md`):

| Semantic | Status under source-backed fulfillment |
|---|---|
| Grant enforcement, projection, revocation, record identity, structured errors | Unchanged and mandatory; all defined on RS *output*, agnostic to record origin. |
| Exact scalar filter, `fields`/`view`, opaque cursor pagination, stable `(cursor_field, primary_key)` sort | Mandatory; the adapter must produce them or the request is rejected. Opaque cursors may wrap upstream pagination state only under the binding rules in "Pagination consistency" below. |
| Range filters, expand, search, aggregation | Declaration-driven; a source-backed stream declares only what it can honestly serve. |
| `changes_since` on `mutable_state` streams | Tier-1 MUST with projection-scoped eligibility ("returning a record whose authorized projection is unchanged is a protocol violation"). Not satisfiable without local version history. **Therefore v1 restricts source-backed mode to `append_only` streams** — the mutable case requires a local projection/version ledger and is deferred. |
| Freshness | MAY at Core, MUST at Collection Profile Tier 2; source-backed responses always carry it. |
| Audit/provenance | Implementation-defined at Core; this change makes upstream fetches spine-auditable. |
| Availability under upstream failure | Unspecified in Core (deferred as "Request-Side Freshness Requirements"); this change stays response-side only, per that deferral's explicit sequencing. |

## Pushdown-or-refuse, and where fetch-and-compensate is allowed

Prior art splits: Trino/FDW default to silent compensation (fetch broadly, filter locally); Steampipe fails loudly on missing required quals; BigQuery's federated `ORDER BY` silently vanishing is the canonical failure. For grant-scoped personal data, the default is **refuse loudly**:

- At the client boundary, capability is binary: declared (works) or undeclared (400). No silent approximation, ever.
- Within a declared shape, the adapter may push the predicate upstream or fetch-and-locally-filter, under two bounds: the upstream fetch envelope is derived from the effective (grant ∧ request) filter and capped by declared page/window/overscan limits; and results must still satisfy the stable-sort contract or the request is rejected.
- The precise privacy accounting: grants govern disclosure at the server→client boundary, not the server's own acquisition (the Collection Profile already fetches broadly with owner credentials). What is new here is that a *client's request* now causes upstream fetching — so the envelope bound exists to stop a client from driving unbounded upstream acquisition, rate-budget exhaustion, or cache-residue accumulation through the personal server. Over-fetch within the envelope lands only in bounded non-canonical cache and is never disclosed beyond the grant projection.
- The Core base query surface is never conditionally rejected. Core's durable base surface (limit/cursor/order/exact filter/fields/view, detail reads) carries no clause permitting a served stream to refuse a bare list, and a hidden posture change must never invalidate existing grants or clients — so eligibility requires the adapter to serve the full base surface, with pagination as the boundedness mechanism: each page's upstream work is capped by page size plus overscan, and sustained deep pagination is governed by Core rate limiting (429 + `Retry-After`), not refusal. Steampipe-style required qualifiers (required KeyColumns, OData `RequiredProperties`) are therefore NOT a v1 mechanism: an upstream that genuinely cannot enumerate in bounded pages makes the stream ineligible rather than conditionally admitted. Required-filter admission remains deferred research for a future version (it would need Core-level semantics, since it changes what a valid base request is).

## Minimal local state

Always present (not record retention): connection credential, manifest/schema, grants, audit spine, upstream sync/window state. Conditionally justified, each tied to a semantic:

- **Bounded response cache** — latency/rate protection; non-canonical, TTL'd, flushable, never widens a response (spec'd).
- **Stable-ID dependence** — record identity comes from the source (already true at ingest: Gmail IMAP UID, Slack ts). If an upstream lacks stable IDs, the stream is not source-backed-eligible; identity synthesis would require materialization anyway.
- **Version/projection ledger** — only needed for `mutable_state` `changes_since`; excluded from v1 by the append-only restriction.
- **Search/semantic indexes** — only needed if search affordances are declared; v1 source-backed streams do not declare them. Index-over-remote-bodies is a legitimate future posture on the spectrum, not v1.

Storage-abstraction note: `design-notes/broad-storage-abstraction-2026-04-24.md` defers a broad storage interface until a concrete trigger. This change deliberately does **not** trip that trigger: the integration seam is the host-injected `queryRecords` operation dependency (`operations/rs-records-list/index.ts`), which MCP already proves is transport-agnostic — a fulfillment dispatch above storage, not a new storage API. The one-method `StorageBackend` pilot's *methodology* (narrow interface, dual conformance harness) is reused; its shape is not.

## Mixed connections

Fulfillment posture is per stream per connection. The pilot itself is mixed: retained `messages` beside source-backed `message_bodies`. V1 confines `expand[]` on an actively source-backed stream to relations the adapter can hydrate within the same fetch envelope (declared in the source-backed `query` subset); cross-posture expansion is not declared in v1. Mixed *within* a stream (imported history + live tail) is the bulk-import note's cursor-alignment problem — explicitly out of scope, but the `origin` provenance vocabulary is shared so the two designs compose later.

## Capability is static; policy is per connection (the load-bearing split)

Two different facts must never share one field:

- **Capability** — "this connector's adapter *can* serve this stream source-backed, under these constraints" — is a static property of connector code. It lives in the manifest as an optional per-stream `fulfillment.source_backed` object (eligibility contract including base-surface support, per-page envelope bounds, and the `query` subset) and is validated deterministically from the manifest JSON alone, like every other manifest capability.
- **Policy** — "this connection *is currently* serving this stream source-backed" — is an owner decision about a configured connection. It lives in connection configuration as a per-stream posture selection (`retained` default, `source_backed` selectable only where capability is declared), is validated against the manifest deterministically (active posture = pure function of manifest + connection configuration), and posture changes are spine-recorded owner actions.

A static manifest cannot encode a connection-specific choice — the same Gmail manifest serves an owner who wants a full retained archive and an owner who wants zero retention. This mirrors the repo's settled catalog-vs-connection lifecycle split (`design-notes/connector-catalog-vs-connection-lifecycle-2026-06-02.md`): the manifest is the catalog fact, the connection row is the lifecycle fact. Clients are insulated from both: posture is not a request, grant, or consent parameter; clients see only declared query capability, freshness, provenance, and structured failure.

The same split resolves the effective-capability problem: retained/index-backed serving may support search, aggregation, and the full filter surface while the live adapter supports a subset, so one static `stream.query` declaration cannot describe both without over-advertising one mode or crippling the other. The source of truth is therefore two declarations in one grammar: the stream-level `query` surface (governs `retained`) and `fulfillment.source_backed.query`, validated as a strict subset of it (governs `source_backed`). The active posture selects which one is effective, and connection-scoped discovery (`schema(stream)`, `field_capabilities`, `expand_capabilities`) advertises only the effective capability — an agent inspecting a connection never sees an affordance the active posture cannot honor, and a posture switch visibly changes the advertisement (conformance-tested both directions). No parallel vocabulary is introduced; the subset reuses the existing declaration grammar, which is the OData `*Restrictions` insight (per-collection, per-operation declared capability) applied with PDPP's stricter default of not-declared-means-not-supported.

Clients do not negotiate freshness (request-side freshness is explicitly deferred in `spec-deferred.md`; this design stays response-side). The read dispatch is a table lookup — active posture + declared capability + cache state — not a cost-based planner. That is a deliberate anti-goal: a second query engine is the failure mode this design exists to avoid.

Posture also governs the collection lifecycle, or the feature would not deliver its stated value: an actively source-backed stream is excluded from scheduled/manual collection scope and ordinary canonical ingest (explicit owner import/migration excepted), so retention actually stops rather than merely being bypassed at read time. Pre-existing canonical rows go dormant — preserved, never auto-deleted on a posture switch, not served, not counted as coverage, not advertised — until the owner explicitly deletes them. Switching back to `retained` is a catch-up state (dormant rows served with `stale`/`unknown` freshness), never instant `current`, until collection re-establishes coverage evidence. This reuses the existing coverage/freshness evidence machinery rather than adding a new health vocabulary, with one explicit addition so health stays honest in both directions: collection reports carry an accepted-not-collected `source_backed` disposition, the health projection treats collection coverage as not-applicable for the posture (readiness follows credential/adapter or probe evidence, never collection recency), and owner surfaces label the stream served-on-demand while disclosing dormant retained size with a separate explicit delete — a mixed connection can be green without pretending the source-backed stream was collected, and without hiding that dormant rows still exist physically.

## Pagination consistency against a moving upstream

Live upstreams change between pages. The contract (spec'd): source-backed page cursors are opaque logical keyset cursors over the Core stable sort `(cursor_field, primary_key)`, bound to the effective filter, projection, order, connection, and an upstream consistency marker (source-generation/session-validity signal — e.g. IMAP `UIDVALIDITY` — plus credential generation). Keyset pagination over an append-only stream is inherently insert-tolerant (a fixed boundary key neither duplicates nor skips already-read positions); what it cannot survive is upstream renumbering or session invalidation, so those fail loudly as Core `invalid_cursor` (restart) rather than paginating with silent gaps — the BigQuery dropped-`ORDER BY` lesson applied to cursors. Upstream pagination tokens may be wrapped only when the upstream's order *is* the declared keyset order; otherwise the adapter must produce keyset pages itself or the shape is not declared.

## Why the manifest surface earns its cost

One optional per-stream capability block (`fulfillment.source_backed` with per-page envelope bounds and a `query` subset) plus one per-stream connection selection, reusing the existing `query` declaration grammar for the query surface. Justification: (a) manifest validation is deterministic-from-JSON by construction in this repo — capability evidence derived dynamically from adapters cannot be validated that way, cannot inform catalog/consent surfaces before a connection exists, and drifts silently; (b) the repo's standing construction rule requires advertised capabilities to agree end-to-end (schema → docs → tool input shape → route → enforcement) with positive and negative conformance gates, which needs a declared source of truth; (c) OData's Capabilities vocabulary and Steampipe's KeyColumns are precedents that per-collection, per-operation declared capability is the honest shape. Rejected alternative: probing adapters at connection time and caching discovered capability — fails (a) and (b), and Google can change upstream behavior under a probe just as easily as under a declaration; declared capability plus conformance tests is at least auditable.

## Alternatives considered and rejected

- **Client-visible fulfillment-mode enum in grants or requests** — rejected. Capability + provenance + freshness carry all decision-relevant information; a mode enum would leak implementation posture into consent semantics and multiply the promise surface.
- **Unbounded fetch-and-filter compensation (Trino default)** — rejected for the client boundary; allowed only inside declared envelopes. BigQuery's silently-dropped `ORDER BY` is the cautionary tale.
- **"Passthrough" as the durable name** — rejected (MCP token-passthrough collision; repo-internal overloads).
- **Core spec change now** — rejected. Under the wire-interop test (`docs/research/collection-layer-boundary-note.md`), nothing here changes what a conformant client must implement: `meta.fulfillment` is additive, `freshness` is existing Core vocabulary, `source_unavailable` is a new error code following the RI-beyond-spec disclosure discipline. Core promotion becomes worthwhile when independent implementations need to interoperate on fulfillment disclosure — tracked as a residual, not done speculatively.
- **`coverage_policy: "live_on_demand"` instead of a `fulfillment` block** — rejected. `coverage_policy`/coverage strategies describe *collection* evidence; overloading them would conflate "what has been collected" with "how reads are served," exactly the conflation the freshness/coverage vocabulary was built to avoid.
- **Browser-automation live reads** — rejected. Polyfill browser connectors carry bot-detection and latency characteristics incompatible with request-time fetching; source-backed mode presumes a cooperative API/credential surface (the collection-prior-art trust-boundary analysis anticipated this).
- **Delegated grant enforcement to an upstream PDPP server (`provider_native` federation)** — deferred, unchanged from the design note: grant enforcement stays local unless delegation is explicitly specified some day. `provider_native` remains the adjacent precedent (the platform itself is the RS), not this feature.
- **`mutable_state` live fulfillment via projection-hash ledger** — deferred, not rejected. It is the known path to lifting the append-only restriction; it needs its own design (ledger residency, retention, cursor expiry) and a real consumer.

## The two counterexamples, answered

**Gmail (strong credential).** Materially helped — but per stream, per the manifest's actual semantics, not as a slogan. Ground truth: `messages`, `threads`, and `labels` are declared `mutable_state` (flags and labels change), so v1 forbids serving them source-backed; they stay retained with version history, which is exactly what their Tier-1 `changes_since` obligation requires. `message_bodies` and `attachments` are `append_only` — immutable payloads, and the bulky, expensive-to-retain part of a mailbox. The pilot is therefore a **mixed connection**: thin retained `messages` metadata (indexed, searchable, change-tracked as today) plus source-backed `message_bodies` — bounded text/HTML payloads served as ordinary Core reads: bare list pages by `(cursor_field, primary_key)`, exact-filter and detail reads by id, each page a bounded IMAP fetch window. No conditional admission is involved; grants MAY further scope reads via `resources`/`time_range`, but an unconstrained list simply pages from the newest bodies down, bounded per page. `attachments` stays out of the pilot even though it is `append_only`: its records carry `blob_ref`, and avoiding retention of attachment bytes would require a source-backed blob lifecycle (identity, authorization, proxy/redirect, cache, retry) this change does not define; the spec makes `blob_ref`-bearing streams ineligible in v1 and defers blob fulfillment to a separate earned extension. Attachment metadata may remain retained. The fetch path is the *existing* IMAP app-password adapter — a `UID FETCH` of a known message is a cheap bounded round trip, and `UIDVALIDITY` is the cursor consistency marker — not a hypothetical Gmail REST credential that is not wired today (the Gmail API's bounded list/search + detail-fetch shape would slot into the same contract if an API-credential adapter is ever added). One eligibility fact the validator enforces: `message_bodies` currently declares no `cursor_field`, so the implementation tranche adds one before the stream can be selected — the honesty mechanism working as designed. What still forces local processing: search affordances (index-backed by construction), label/flag change tracking, and cold full-history enumeration. The win is real and specific: no retention of large immutable message bodies while metadata stays indexed.

**Slack (30–45 minute incremental archive refresh).** Not helped today, and the design says so rather than pretending. The retained verdict rests on the connector's shape and the upstream's query surface: a `slackdump` subprocess replaying a captured browser-session token, 6–20 hour cold archives, and no upstream ordering/projection semantics compatible with the read contract. Slack's May 2025 rate-limit change (1 request/minute, 15 objects, for newly created/new-install commercially distributed non-Marketplace apps; internal customer-built apps explicitly retain higher limits) is a constrained portability-market signal — it neither governs the owner's current session-token path nor generalizes to "sources penalize sync-and-store" — and it neither forces nor enables live reads here. Slack stays `retained` with its incremental refresh loop, which its manifest already encodes (`recommended_interval_seconds: 1800`). The honest statement: source-backed fulfillment reduces Slack's work by zero right now; the trigger to revisit is a cooperative Slack query API reachable with a durable credential.

## Acceptance checks

1. `openspec validate define-source-backed-fulfillment --strict` passes.
2. Manifest validator: accepts `fulfillment.source_backed` (with a valid `query` subset) on an eligible `append_only` stream; rejects it on `mutable_state`, on `blob_ref`-bearing streams, on missing deterministic `cursor_field` or missing base-surface support, and on a `query` subset exceeding the stream surface; absent block leaves behavior byte-identical (existing manifest corpus revalidates unchanged).
3. Connection selection: selecting `source_backed` where the capability is declared activates the posture and emits a spine event; selecting it where undeclared is rejected with a structured error; a connection with no selection behaves byte-identically to today.
4. Effective-capability pair: with posture `retained`, a declaration-driven retained-only shape (e.g. lexical search or an undeclared-in-subset range filter) succeeds and is advertised in `schema(stream)`; after switching to `source_backed`, that shape disappears from connection-scoped discovery and returns HTTP 400, while every base-surface request and every subset shape still succeeds; switching back restores the original advertisement and behavior.
5. Conformance pair per the repo's standing rule — canonical positive: subset-declared filter on an actively source-backed stream returns live-fetched, grant-projected records with `meta.fulfillment.origin: "live"` and fresh `captured_at`; plausible negative: undeclared filter shape returns HTTP 400 with no upstream fetch issued (assert via spine: `fetch_attempted: false`).
6. Cursor honesty: multi-page pagination with upstream mutation between pages yields no gaps or duplicates for keyset pages; a changed upstream consistency marker (e.g. `UIDVALIDITY` rotation) or mismatched effective-filter binding yields 400 `invalid_cursor`.
7. Unavailability: upstream fault injection yields 503 `source_unavailable`, never an empty 200; with warm cache inside the staleness bound, yields `origin: "cache"` with honest freshness.
8. Token boundary: assert no client or owner token material in any upstream request issued by the fixture adapter. Owner self-export against an actively source-backed stream works under the same per-page-bounded contract.
9. Cache isolation: an entry cached under one grant/connection/credential generation is not served under another; credential rotation and posture change invalidate; flush mid-session forces refetch; coverage/freshness evidence for retained streams unchanged throughout.
10. Pilot (real source, owner-gated, dev instance only): mixed Gmail connection — `messages` retained; `message_bodies` (after gaining its `cursor_field`) selected source-backed and served through the existing IMAP adapter as bounded list pages and detail reads, including a bare unfiltered list; attachment metadata retained, attachment bytes untouched (blob fulfillment deferred); MCP `query_records`/`fetch` over the same grant works with zero MCP-layer changes and discovery shows the effective capability only.

## Residual risks

- **Upstream drift**: declared capability can go stale when the source changes behavior (OData's model never faced this; we wrap sources we don't control). Mitigation: conformance fixtures per adapter and `source_unavailable`/validation telemetry; drift detection beyond that is future work.
- **Rate multiplexing**: many grants share one upstream credential's rate budget; v1's per-request envelope bounds are a floor, not a governor. The provider-rate-governance work (`docs/research/provider-rate-governance-convergence-2026-06-10.md`) is the natural home for the full answer.
- **Latency**: request-time upstream fetches put source latency on the client's critical path; `freshness`/`origin` disclose it, and the cache bounds it, but v1 sets no latency SLO.
- **Core promotion timing**: if a second implementation adopts fulfillment disclosure, `meta.fulfillment` and `source_unavailable` need Core-level advertisement (`capabilities.*` pattern per the spec-readiness audit) before interop claims.
