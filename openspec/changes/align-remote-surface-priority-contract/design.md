## Context

Remote-surface 1.x makes lease priority host-neutral: `interactive` outranks `background`. PDPP previously encoded manual/scheduled posture in the priority value even though `triggerKind` already records manual, scheduled, and webhook causes. Browser-surface lease rows persist the old values in SQLite and Postgres.

## Goals / Non-Goals

**Goals:**

- Use the 1.x priority vocabulary at every PDPP resource-priority boundary.
- Preserve the ordering (`interactive` before `background`) and existing trigger-kind behavior.
- Upgrade existing SQLite and Postgres rows safely at normal startup.
- Prove, through the installed package, that retained surfaces are not routine idle-TTL or capacity-pressure reap candidates.

**Non-Goals:**

- Pinning an obsolete 0.x package or retaining a bidirectional priority adapter.
- Renaming `triggerKind`, changing scheduler/manual/webhook policy, or changing the durable run trigger values.
- Changing the live stack or publishing a package.

## Decisions

### One-way vocabulary migration

PDPP accepts and persists only `interactive` and `background`. `triggerKind` continues to represent `manual`, `scheduled`, or `webhook`, because urgency and cause are independent facts. This removes the misleading host-specific names rather than hiding them behind a permanent adapter.

### Startup schema migration is catalog-gated and fail-closed

SQLite inventories the lease table's columns, foreign keys, explicit indexes, and triggers before rebuilding it inside one transaction. It recognizes only the historical direct enum-check shapes, maps legacy rows with an explicit `CASE`, recreates PDPP and preserved dependent objects before `foreign_key_check`, and rolls the whole operation back on an unsupported shape or copy failure. A mixed old/new priority check is migrated; completion requires the exact new-only check and no legacy row.

Postgres begins a dedicated transaction-scoped advisory-lock transaction before it discovers or classifies priority constraints and rows. A waiting starter therefore rereads the first starter's committed current schema before deciding whether any priority DDL is necessary. It replaces only the known direct legacy named or autogenerated shape, leaves unrelated named and unnamed compound checks alone, and maps both legacy values atomically. A current direct check takes no priority DDL path on later boots and the advisory lock does not request an `AccessExclusiveLock` on the lease table. The public bootstrap uses a separate session advisory lock with bounded application-side exponential-backoff polling via `pg_try_advisory_lock` on the checked-out client. Each failed lock attempt completes before sleeping, so it cannot leave a waiting virtual transaction that deadlocks either lexical-GIN or hot-HNSW `CREATE/DROP INDEX CONCURRENTLY` work in the lock holder. The same client unlocks in `finally`; the fallback serializes pre-existing additive migrations without a table lock or schema redesign. Lifecycle check upgrades are likewise catalog-gated so current constraints are not churned.

### Published package is the retention oracle

The PDPP boundary test creates retained surfaces through the installed `1.5.1` `/leases` package and proves idle cleanup and capacity-pressure planning do not select them. This release gate is green; PDPP does not add an adapter or pin an obsolete package.

## Risks / Trade-offs

- [An interrupted or concurrent schema migration] → SQLite rebuild runs in its existing startup retry path and transaction; Postgres polls for the bootstrap session lock without holding a blocking SQL statement, then takes a transaction-scoped migration advisory lock before catalog discovery and performs the targeted priority constraint replacement and data mapping atomically.
- [A legacy priority is missed] → post-migration checks reject old values and tests assert both legacy-row mapping and new-only constraints.
- [A published package regresses retained-surface protection] → the installed-package boundary test fails before PDPP can claim the release handoff is complete.

## Migration Plan

1. Update runtime types, defaults, and callers to use `interactive`/`background` without changing `triggerKind`.
2. Upgrade persisted rows and recognized historical constraints at startup; a successful restart is safe to repeat because exact new-only schemas no longer trigger conversion. Unknown lease check shapes fail closed rather than being guessed.
3. Regenerate the lockfile so both importers resolve `^1.5.1` exactly to published `1.5.1`, then run the installed-package boundary test.
4. Rollback before deploying is a code/package rollback; once a database has migrated, old code cannot safely write legacy values and must not be redeployed.

## Open Questions

None. The published `1.5.1` retained-surface boundary is green.
