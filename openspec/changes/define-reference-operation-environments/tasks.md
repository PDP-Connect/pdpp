## 3. Proof Slice Planning

- [x] 3.1 Define storage-only security proof: `ConsentStore` + `OwnerDeviceAuthStore` with SQLite and memory adapters. (Prerequisite harness landed via `add-consent-device-auth-conformance-harness`: SQLite driver + deliberately-broken in-memory falsifiability driver + 12 lifecycle/security scenarios. Conforming second adapter landed via `reference-implementation/test/helpers/memory-consent-device-auth-driver.js` and `reference-implementation/test/consent-device-auth-conformance-memory.test.js`: the same harness now runs green against SQLite *and* a conforming in-memory driver, while still failing loudly against the broken falsifiability driver. The storage-only adapter-swap proof is therefore complete at the test level. No production `ConsentStore` / `OwnerDeviceAuthStore` interface is extracted by this slice; that promotion remains a separate change.)
- [x] 3.2 Define Postgres-oriented storage proof: `ConnectorStateStore` + `SchedulerStore` with SQLite and a non-default Postgres adapter spike. (Prerequisite harness landed via `add-connector-state-scheduler-conformance-harness`: SQLite driver + broken driver covering connector-state, schedule, and active-run invariants. The non-default Postgres adapter spike now lands as a dev-scoped, env-gated test driver: `reference-implementation/test/helpers/postgres-connector-state-scheduler-driver.js` reimplements the three concerns directly against Postgres (no SQLite reference coupling) with its own DDL — partial unique indexes for owner-vs-grant state scope, `ON CONFLICT (connector_id) DO UPDATE` for schedules and active-run upsert, `UNIQUE(run_id)` for cross-connector run_id identity, and a transactional drain-into-`terminal_failed_runs` reconciliation in `simulateRestart`. `reference-implementation/test/connector-state-scheduler-conformance-postgres.test.js` runs the full 16-scenario harness when `PDPP_TEST_POSTGRES_URL` is set, and registers a single skipped test otherwise so the suite acknowledges the proof exists without requiring Postgres in default test environments. Self-falsified twice on a real Postgres 16 instance — dropping `UNIQUE(run_id)` causes the cross-connector run_id scenario to fail; collapsing grant_id on read causes both grant-isolation scenarios to fail; restoring each makes all 16 scenarios pass. The `pg` package is dev-scoped on `reference-implementation`; no production Postgres adapter is introduced. Promoting these conformance shapes into a production `ConnectorStateStore` / `SchedulerStore` interface remains a separate change.)
- [x] 3.3 Define sandbox operation proof: `rs.streams.list` mounted through Fastify and Next sandbox, with `buildLiveStreamsList` deleted. (Landed as `mount-rs-streams-list-operation`. Extended in the same idiom by `mount-rs-stream-detail-operation` and `mount-rs-schema-get-operation`; `buildLiveStreamsList`, `buildLiveStreamMetadataResponse`, and `buildLiveSchemaResponse` are deleted from `apps/web/src/app/sandbox/_demo/builders.ts`.)
- [x] 3.4 Define import-boundary checks or grep-based gates that prevent operations from importing concrete SQLite/Fastify/Next/process dependencies. (Generalized discovery-based gate landed via `add-reference-operation-boundary-gate`: a shared helper at `reference-implementation/test/helpers/operation-boundary.js` and `reference-implementation/test/operations-boundary.test.js` discover every module under `reference-implementation/operations/*/index.ts` and assert no static imports of host, storage, sandbox, or Node `process` modules and no executable `process.env` access. The per-operation boundary tests now consume the same shared helper. Dynamic imports are explicitly out of scope.)
- [x] 3.5 Define conformance/equivalence tests that must pass against the existing local server and the candidate sandbox/test host. (Five conformance harnesses landed: consent/device-auth, connector-state/scheduler, disclosure-spine, record-mutation, record-read. Each ships SQLite and falsifiability/broken drivers. Sandbox-vs-native equivalence is asserted at the operation level for `rs.streams.list`, `rs.streams.detail`, and `rs.schema.get`.)
- [ ] 3.6 Identify rollback criteria and stop conditions for each proof slice. (Design.md "Records/Search Feasibility Gate Result" enumerates stop conditions for the records/search feasibility gate. Per-slice rollback criteria — e.g. when to revert a host mount, when to back out a conformance harness — are not written down explicitly.)
