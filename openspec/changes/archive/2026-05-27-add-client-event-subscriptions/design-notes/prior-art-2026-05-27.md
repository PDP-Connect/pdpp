# Prior art: outbound webhook / event-subscription systems

_Compiled 2026-05-27. Sources fetched live; URLs at the end of each section are
the exact docs reviewed._

PDPP's "client event subscriptions" surface needs: a personal data server
notifies authorized clients (apps/AI agents) that grant-visible data changed;
the client then fetches via existing read APIs. The relevant design decisions
are: envelope shape, signing, timestamp tolerance, idempotency, retries,
auto-disable, subscription handshake, hint-vs-payload, and
renewal/expiration.

## Per-system design extracts

### Stripe Webhooks
- Envelope: data lives in the same JSON object as routing metadata (`type`, `id`, `created`, `livemode`, `data.object`).
- Signature is over `f"{t}.{body}"`; `Stripe-Signature` is a CSV of `t=`, `v1=`, optional historical `v0=`. Multiple `v1=` values allowed during secret rotation.
- Idempotency: log `event.id`. Cross-warning that Stripe sometimes emits two distinct Events for the same change → dedupe on `(event.type, data.object.id)` too.
- Retry curve: exponential, "up to three days" live; manual resend doesn't cancel auto retries.
- Thin events are the strategic direction: tiny, unversioned, type-safe; client uses SDK `fetchRelatedObject()`. This is the closest mainstream analog to PDPP's "hint + fetch via read API" idea.
- Docs: <https://docs.stripe.com/webhooks>, <https://docs.stripe.com/event-destinations>

### GitHub Webhooks
- No automatic retries. Delivery list + manual `redeliver` REST endpoint is the durability story. Consumers run a polling loop over the delivery API as a safety net.
- 10-second response window; `X-GitHub-Delivery` UUID is the canonical idempotency key, separate from event payload IDs.
- `ping` event verifies endpoint on creation.
- Docs: <https://docs.github.com/en/webhooks/using-webhooks/validating-webhook-deliveries>, <https://docs.github.com/en/webhooks/using-webhooks/handling-failed-webhook-deliveries>

### Plaid Webhooks
- Verification via JWS (ES256) carried in a `Plaid-Verification` JWT whose payload contains `request_body_sha256`. The asymmetric key story is operationally heavier than HMAC but supports key rotation without sharing secrets.
- Codes (`SYNC_UPDATES_AVAILABLE`, `HISTORICAL_UPDATE`, `DEFAULT_UPDATE`, `INITIAL_UPDATE`) are opaque hints; the durable pull path is `/transactions/sync` with a client-held cursor — webhook never carries the data.
- Auto-disable: if >90% of webhooks in 24h are rejected, retries stop. Strong precedent for failure-rate-based circuit breaking.
- Explicitly tells integrators to expect duplicates and out-of-order delivery and to design idempotent handlers.
- Docs: <https://plaid.com/docs/api/webhooks/>, <https://plaid.com/docs/api/webhooks/webhook-verification/>, <https://plaid.com/docs/api/products/transactions/>

### Google push notifications (Drive/Calendar watch channels)
- Empty body, headers-only. This is the cleanest "pure hint" implementation in the wild — the protocol structurally cannot leak data.
- Pre-shared `token` (echoed as `X-Goog-Channel-Token`) substitutes for cryptographic signing — much weaker than HMAC, but simpler.
- No renewal API. The model is: watch with explicit expiration → client re-watches with a fresh channel id before expiry → temporarily two channels deliver overlapping events.
- `sync` message with `X-Goog-Message-Number: 1` is a built-in reachability check.
- Docs: <https://developers.google.com/workspace/drive/api/guides/push>

### Microsoft Graph change notifications
- Two-step subscription handshake: POST creates subscription → Graph sends GET-like POST with `validationToken` query param → endpoint must respond `200` with the token in body within 10 seconds.
- `clientState` is a per-subscription shared secret returned in every payload; weak but allows the receiver to bind a notification to a subscription it created.
- Three notification flavors:
  - **Basic**: id + resource path → client fetches via Graph.
  - **Rich**: payload encrypted with subscriber's public key (asymmetric).
  - **Lifecycle**: `subscriptionRemoved`, `missed`, `reauthorizationRequired` — explicitly tells client to reconcile or re-auth.
- Per-resource `expirationDateTime` ceiling (3 days for chat, 30 days for Drive, etc.); subscriber PATCHes to renew.
- Docs: <https://learn.microsoft.com/en-us/graph/change-notifications-overview>

### W3C WebSub
- Subscription verification is GET-based: hub hits `hub.callback?hub.mode=subscribe&hub.topic=…&hub.challenge=<random>&hub.lease_seconds=…` and the subscriber echoes `hub.challenge`. Confirms callback URL ownership.
- HMAC of content body with `hub.secret` in `X-Hub-Signature: sha256=<hex>` — algorithm prefix in the header value, mirroring GitHub.
- `lease_seconds` is subscriber-requested but hub-enforced; subscriber re-subscribes before expiry.
- Spec mandates HTTPS and unguessable callback URLs as the baseline against spoofed subscriptions.
- Docs: <https://www.w3.org/TR/websub/>

### CloudEvents 1.0
- Useful as an _envelope spec_, not a transport. Defines `id`, `source`, `type`, `specversion` as required; `time`, `subject`, `data`, `datacontenttype` optional.
- Dedupe rule is normative: `source + id` uniquely identifies an occurrence. Critical because per-system event IDs aren't globally unique.
- HTTP binding: **binary mode** maps each attribute to a `ce-<name>` header and keeps `data` raw in the body; **structured mode** puts the whole event JSON in the body. Binary mode aligns with Stripe/GitHub-style transport headers; structured mode aligns with MS Graph-style envelope-in-body.
- Docs: <https://github.com/cloudevents/spec/blob/v1.0.2/cloudevents/spec.md>

### MCP resource subscriptions
- `notifications/resources/updated` carries **only** the resource `uri`. Client then issues `resources/read`. Closest direct analog to PDPP's intent.
- Subscription via in-session `resources/subscribe` request, confirmed by JSON-RPC response (the session itself is the trust boundary; no per-message HMAC).
- `notifications/resources/list_changed` is a separate signal for catalog churn (new/removed resources) vs. content changes. PDPP should consider the analogous split: "what you can see changed" vs. "a thing you can see changed".
- Docs: <https://modelcontextprotocol.io/specification/2025-06-18/server/resources>

## Synthesis: what PDPP should adopt

1. **Envelope = CloudEvents 1.0 in HTTP binary mode.** Use `ce-id`, `ce-source`, `ce-type`, `ce-specversion`, `ce-time`, `ce-subject` headers; body holds the hint payload. Free deduplication semantics, free interop with EventBridge / Event Grid / Knative consumers.
2. **Pure-hint payloads by default**, MCP/Plaid/Google-style. Body contains `{grantId, resourceUri, changeType, cursor?}` only — never the changed data. Forces all access through the existing scoped read API, keeping authorization centralized.
3. **HMAC-SHA256 signing with timestamp**, Stripe-style. Header form: `PDPP-Signature: t=<unix>,v1=<hex>` over `t + "." + body`. Constant-time compare. Allow multiple `v1=` values during secret rotation.
4. **Five-minute timestamp tolerance default**, configurable, never zero. Matches Stripe; defends against replay without breaking real-world clock skew.
5. **Idempotency on `(ce-source, ce-id)`** as the normative pair, with a secondary advisory composite of `(grantId, resourceUri, ce-type)` to catch the "two events, same change" case Stripe warns about.
6. **Retry policy: exponential backoff up to 24 hours**, Plaid-style. Respect `Retry-After`. Persist the delivery attempt log so clients (and PDPP support) can inspect history — this is what GitHub gets right where Stripe doesn't surface as clearly.
7. **Failure-rate auto-disable**, Plaid-style: if >90% of attempts in a rolling 24h window fail, pause the subscription and emit a lifecycle event. Tunable per-grant. Avoids Stripe's "we may disable you" opacity.
8. **Lifecycle events as a first-class type**, MS Graph-style. Emit `pdpp.subscription.paused`, `pdpp.subscription.reauthorization_required`, `pdpp.subscription.missed_events` over the same transport. Clients implementing one handler can't accidentally miss operational signals.
9. **Subscription verification handshake on registration**, WebSub/Graph-style. PDPP POSTs to the client callback with a `validationToken` (or GETs with `challenge` — pick one; WebSub-style GET is simpler). Reject the subscription if the callback doesn't echo. Cheap proof-of-possession of the URL.
10. **Explicit expiration + client-driven renewal**, Graph/WebSub-style. Cap subscription lease at e.g. 30 days. Client PATCHes to renew. Forces grant-lifecycle alignment — a revoked grant naturally drains.
11. **Two notification axes**, MCP-style: `pdpp.resource.updated` (content of a known resource changed) vs. `pdpp.grant.scope_changed` (set of visible resources changed). Don't conflate.
12. **Cursor included in the hint**, Plaid-style. Hint says "new data available since cursor X"; client passes cursor back to the read API. Removes the need for the client to scan since-last-fetched timestamps and naturally handles missed-event reconciliation after pause/resume.
13. **Initial `sync` ping**, Google-style. First message on a new subscription is a no-op delivery with the subscription's own metadata — proves the channel works end-to-end before real events flow.
14. **HTTPS-only callbacks, unguessable per-subscription URL paths recommended**, WebSub-style — defense in depth on top of HMAC.
15. **Spec the dedupe + idempotency requirement on the client side**, Plaid-style. Document it loudly: "you WILL receive duplicates and out-of-order events; your handler MUST be idempotent." Avoids the false sense of exactly-once.

## What PDPP should NOT adopt

- **Stripe-style "thin" SDK fetch pattern requiring a Stripe-shaped client object graph.** Too SDK-bound; PDPP's hint should be self-contained enough to drive the existing read API without a special SDK.
- **GitHub-style no-retry.** Personal data is too important to leave durability to a polling loop the client has to write. Always retry server-side.
- **GitHub `X-Hub-Signature-256` legacy SHA-1 fallback header.** Don't ship two algorithms; ship one.
- **Google-style pre-shared opaque token instead of HMAC.** Too weak. Take Google's header-only envelope shape, but keep HMAC signing.
- **Google-style "no renewal endpoint, just re-watch with a new id."** Forces client-side overlap handling. Prefer the Graph/WebSub PATCH-to-extend model.
- **MS Graph rich-encrypted-payload mode.** Asymmetric per-subscription key management is operationally heavy and contradicts the "pure hint, fetch via read API" design choice. Keep payloads as hints; let the read API enforce confidentiality.
- **WebSub hub-side multi-algorithm support (SHA-384/512).** YAGNI; pick SHA-256.
- **Plaid-style JWS (ES256) signing.** Strong, but the asymmetric key fetch + key-rotation lookup is a lot of moving parts for a v1. HMAC with a Stripe-style rotation window is enough.
- **CloudEvents structured-mode** (whole-event-in-body). Binary mode keeps headers as headers and bodies as bodies — better for log inspection, signing, and webhook UIs that expect a recognizable JSON body shape.
- **MCP's reliance on a persistent JSON-RPC session for delivery.** Fine for in-process AI agents, wrong for general client/app subscriptions over the public internet. Use HTTP webhooks as the primary transport; offer an MCP-style streaming channel only as a secondary surface.
- **Per-resource subscription quotas that vary wildly (Graph: 1 here, 10 there, 10,000 elsewhere).** Pick a small number of consistent limits per grant; avoid the matrix.
