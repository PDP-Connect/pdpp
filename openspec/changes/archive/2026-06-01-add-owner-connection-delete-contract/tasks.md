## 2. Implementation (lane `ri-owner-delete-connection-implementation-v1`)

- [x] 2.1 Add by-instance, all-streams record-delete queries under `server/queries/records/delete/` (`WHERE connector_instance_id = ?`), parallel to the existing by-stream set; do NOT widen to `connector_id`. (Added `delete-records-by-instance.sql`, `delete-record-changes-by-instance.sql`, `delete-version-counter-by-instance.sql`, `delete-blob-bindings-by-instance.sql`, `delete-blobs-by-instance.sql`, `delete-attention-records-by-instance.sql`, `count-records-by-instance.sql`, `list-streams-by-instance.sql`, plus `device-exporters/clear-source-instance-connector-ref.sql` and `connector-instances/delete-by-id.sql`. Every WHERE clause keys on `connector_instance_id` only.)
- [x] 2.2 Implement `deleteConnection(connectorInstanceId, { ownerSubjectId, now })` on the SQLite and Postgres connector-instance stores: ONE all-or-nothing transactional cascade (I8) keyed on one id (I1), clear `device_source_instances` back-ref, refuse on active-run lease (I7), return a non-secret deletion summary. (Both stores resolve+verify ownership, refuse run-active/default-account, then run the ENTIRE durable cascade in one `writeTransaction` (SQLite) / `withPostgresTransaction` (Postgres): the injected records-family purge (`deleteConnectionRecordRowsSqlite` / `deleteConnectionRecordRowsPostgres(client, …)` from `records.js`) runs INSIDE that same transaction — binding against the same SQLite handle / Postgres client — alongside the schedule delete + device back-ref clear + `connector_instances` row delete LAST. The records purge no longer opens its own transaction, so a failure anywhere in the cascade — record purge OR schedule/device/row cleanup — rolls the whole unit back; there is no half-deleted connection. Stream enumeration is a pre-commit read; the rebuildable lexical/semantic search-index teardown (`teardownConnectionSearchProjection`) runs post-commit and its failure does not invalidate the committed source-of-truth delete. `controller_active_runs` is never erased (an in-flight run is refused, not deleted). Returns `{ connection_id, connector_id, source_kind, deleted_record_count, deleted_stream_count, schedule_deleted, device_refs_cleared }`.)
- [x] 2.3 Implement the Decision-1 default-account no-resurrection guard … OR type default-account delete `unsupported` with the tombstone-gap reason (I6). (Decision-1 FALLBACK chosen for this slice: a default-account binding (`source_kind:'account'`, `source_binding_key:'default'`) is refused with typed `default_account_delete_unsupported` (409) before any mutation, so its deterministic id cannot be hard-deleted and silently re-materialized. The tombstone-ledger construction is deferred; the contract's hard requirement — no deleted connection silently re-materializes — holds because the row is never deleted.)
