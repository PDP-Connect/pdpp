## 3. Public RS Runtime

- [x] 3.1 Thread `connection_id` and `display_name` onto every record-bearing records/search/blob response item. Search hits now carry `display_name` via a per-binding lookup deduped across the snapshot: `server/search.js#buildSnapshot` and `server/search-semantic.js#buildSnapshot` resolve owner-facing labels through the shared `connection-identity.js#resolveDisplayNamesForBindings` helper (one store roundtrip per unique `connection_id` in the snapshot, not one per hit). Hybrid forwards the label from whichever sub-source first emitted it. Placeholder labels (`legacy`, `default_account`, connector-id defaults) are omitted, not faked. Records-list, records-detail, changes_since rows, aggregate-expanded child records, and Postgres-backed record reads continue to carry `connection_id`/`display_name` when the runtime knows the pinned storage binding. Raw blob byte responses are not JSON envelopes; blob identity comes from the authorized record that references the blob. Coverage: `canonical-read-envelope-conformance.test.js` (search display_name emit + omit) plus existing `public-read-connection-id-decoration.test.js` (records).
- [x] 3.2 Make search hits carry `(connection_id, stream, record_id)` without dashboard-side inference. Lexical / semantic / hybrid result items now emit `connection_id` and the deprecated `connector_instance_id` alias when the snapshot recorded the binding. Pre-identity snapshots are tolerated (fields omitted, not faked).
- [x] 3.3 Implement optional `connection_id` filtering and deprecated `connector_instance_id` alias conflict validation on public read routes. Canonical helper `validateConnectionAlias` is shared by `reference-implementation/server/records.js` (records list + detail + aggregate via `validateTopLevelQueryParams`) and the three search operations (`rs-search-lexical`, `rs-search-semantic`, `rs-search-hybrid`). Conflict tests live in `public-read-connection-alias.test.js`. (Filtering itself — narrowing storage to one `connection_id` when supplied — still requires the storage fan-in tranche to land. Today the alias is accepted, deprecated-equality is enforced, but the filter does not yet narrow storage on the records path.)
- [~] 3.4 Normalize public read responses into the canonical envelope, preserving backward-compatible fields only where the contract allows. (Vocabulary-uniform; further reshape deferred.) `finalizeCanonicalEnvelope` at the route layer adds canonical `links.self`, `links.next` (list-shaped only), and defaults `meta.count = { kind: 'none' }` + `meta.warnings = []` for all public read responses while preserving backward-compat fields like `next_cursor`, `next_changes_since`, and `url`. List envelopes (records list, search, stream list) are canonical. Single-record envelopes carry the canonical `object`/`data`/`links`/`meta` vocabulary today — `{ object: 'record', id, stream, data, [identity], links, meta }` — which satisfies design.md ("Single-record, schema, and stream responses use the same `object`, `data`, `links`, and `meta` vocabulary"). Further reshape to a wrapped JSON:API-style single-object envelope (`{ object: 'single', data: { object: 'record', ... } }`) is a wire-breaking change requiring coordinated consumer migration (task 6.2) and is intentionally deferred from this tranche. **Blocker for full reshape:** dashboard/Explorer/MCP consumers still consume the flat `{ id, stream, data }` shape directly; a deeper reshape without coordinated consumer wiring would break dashboards mid-tranche.
- [x] 3.5 Implement strict validation for unsupported parameters, fields, filter operators, sort fields, and expansion targets. Records/search reject unsupported top-level params, unsupported expand relations, conflicting aliases, and unsupported filter operators with typed errors. Canonical `sort` is honored on both SQLite and Postgres backends with sign-prefix direction control and `sort`/`order` disagreement detection. Search ops also reject `count=` and other unsupported parameters at the v1 allowlist. Per-capability `/v1/schema` truth conformance landed in 4.2 (`schema-capability-truth.test.js`).
- [x] 3.6 Implement structured `meta.warnings` for skipped-not-applicable sources, deprecated alias use, count downgrades, and partial/lossy outcomes. Records-list, aggregate, lexical, semantic, and hybrid search emit `deprecated_alias_used` in `meta.warnings[]`; hybrid deduplicates warnings. Lexical and semantic search now also emit `source_skipped_not_applicable` warnings (carrying the connector_id as `source`) when the owner fan-out drops a connector due to a broken manifest or empty searchable plan — previously this was silent. Coverage: `search-source-skipped-warning.test.js`. `count_downgraded` remains wired into the canonical vocabulary and the response plumbing but is intentionally not emitted today: the reference runtime never actually downgrades a count (computing the exact value is cheap, so `count=estimated` returns kind=`exact`), and emitting a downgrade warning on an upgrade outcome would be a wire-shape lie. `partial_results` and `compatibility_fallback` codes remain reserved in the canonical vocabulary for future tranches where the runtime can honestly emit them.
- [x] 3.7 Implement graded count support (`none`, `estimated`, `exact`) backed by existing projections when possible. `count=none` and `count=exact` are fully implemented on both the SQLite reference path and the Postgres runtime path via a visible-row scan over the same filter set the list path uses. `count=estimated` is honored by computing the exact value cheaply on the same scan; the response carries `meta.count.kind = 'exact'` and no warning, since returning a higher-fidelity grade than requested is an upgrade rather than a downgrade per the spec's "Requested count is downgraded" rule. Both backends reject unknown `count` values with `invalid_request`. **Search-path counts are advertised as unsupported** (`count_supported: false` on lexical/semantic/hybrid capability) and the strict allowlist rejects `count=` on `/v1/search*`. This is the honest posture — search runs a ranked snapshot whose total is not knowable cheaply without paging the full snapshot. Promoting search counts to a real grade is a separate runtime change (would require snapshot total-count plumbing) and would belong in a future tranche; it is NOT a blocker on this contract.
- [x] 3.8 Keep one-hop inline expansion grant-safe and bounded by `expand_limit`.

## 4. Capability Document

- [~] 4.1 Update `GET /v1/schema` to be the canonical capability document for streams, fields, operators, sortability, expansion, projection, search modes, pagination, counts, and granted connection identities. **All other capabilities are surfaced today** (`field_capabilities.<field>.exact_filter`, `range_filter.operators`, `lexical_search`, `semantic_search`, `aggregation`, `expand_capabilities`, `cursor_field`, freshness). **Deferred (granted_connections only):** the operation layer already accepts arbitrary per-stream `[extra]` fields, but the host-side `buildConnectorSchemaItem` only knows about a single `storageBinding`. Listing all bindings under a grant per stream requires a new `listGrantedConnections(grant, streamName)` helper on the connector-instance-store and a matching reference-contract schema for `granted_connections: [{ connection_id, display_name }]`. **Blocker:** the storage fan-in tranche (one grant → many bindings) has not landed; until then there is no honest "granted connections" to enumerate per stream — every grant pins one binding. Safe to add once the storage fan-in tranche lands.
- [x] 4.2 Add conformance checks proving every advertised field/operator/sort/expand capability is either enforced or rejected clearly. Coverage: `schema-capability-truth.test.js` exercises the per-field range-operator vocabulary, exact-filter scalar-acceptance, undeclared-field rejection, field-not-granted rejection (grant trumps schema), and the closed range-operator vocabulary against accidentally-advertised unsupported operators. The harness uses the same `manifestStream` shape that feeds both `buildFieldCapabilities` (schema advertisement) and `compileRequestFilters` (runtime validator), so capability drift between the two would fail the harness.
- [x] 4.3 Add search-mode pagination and count-support metadata. `buildLexicalRetrievalCapability` / `buildSemanticRetrievalCapability` / `buildHybridRetrievalCapability` (`server/metadata.ts`) now emit explicit `cursor_supported` and `count_supported` fields. Lexical and semantic advertise `cursor_supported: true, count_supported: false`. Hybrid advertises `count_supported: false` and forwards `cursor_supported`. Runtime allowlists reject `count=` cleanly with `invalid_request`. Coverage: `search-count-capability.test.js`.
